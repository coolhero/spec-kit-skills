# Coverage — SBI Coverage Check & Gap Resolution

> Reference: Read after `/smart-sdd coverage` is invoked.
> For shared rules (MANDATORY RULES, Common Protocol), see SKILL.md.

## Purpose

Checks Source Behavior Inventory (SBI) coverage — how much of the original source code's behaviors are mapped to SDD Features — and optionally resolves gaps by creating new Features or assigning unmapped behaviors to existing ones.

Can be run:
- **Standalone**: `/smart-sdd coverage` anytime to check current coverage
- **From adopt pipeline**: Called automatically as Post-Pipeline Coverage Verification
- **From rebuild pipeline**: Called optionally at pipeline completion

---

## Prerequisite

SBI coverage data must exist in `sdd-state.md` (Source Behavior Coverage section). This section is populated by `/reverse-spec` during source analysis.

If no SBI data exists (greenfield projects), display:
```
ℹ️ No Source Behavior Coverage data found.
SBI tracking is only available for projects with existing source code
(Origin: rebuild or adoption).
```

## Step 2 — Gap Resolution (Interactive)

**If P1 + P2 = 100%**: Display "✅ Full P1+P2 coverage achieved." and stop.

**If gaps exist**, ask what to do:

Use AskUserQuestion:
- **"Resolve gaps now"** — Proceed to Step 2a to classify each unmapped group
- **"View only"** — Stop here (coverage report only, no changes)

**If response is empty → re-ask** (per MANDATORY RULE 1).

> When called from `adopt` pipeline (Post-Pipeline), "Resolve gaps now" is the default and P1+P2 must reach 100% before proceeding. "View only" is not offered.

### Step 2a — Group Classification (HARD STOP per group)

Group unmapped P1/P2 behaviors by their apparent module/Feature affinity (from the SBI's Feature column). For each group, use AskUserQuestion and WAIT for the user's response:

```
📋 Unmapped Behaviors: [Group Name] ([N] items)

  B003 | registerUser()     | P1 — No matching FR in spec.md
  B007 | sendVerifyEmail()  | P2 — No matching FR in spec.md

Options:
```

Use AskUserQuestion per group:
- **"Add to existing Feature [FID]"** — Go back to the affected Feature's specify step and add missing FR-### mappings for these behaviors. Mark the Feature's pipeline steps with 🔀 from specify onward in sdd-state.md. The Feature will re-run the pipeline from specify.
- **"Create new Feature"** — Invoke the `add` workflow with pre-populated definition: Feature name derived from the behavior group, description includes the unmapped B### items as draft requirements, pre-context.md populated with the SBI entries. The new Feature goes through the full pipeline.
- **"Intentional exclusion"** — Record with one of 6 exclusion reasons: `deprecated`, `replaced`, `third-party`, `deferred`, `out-of-scope`, `covered-differently`. Update `coverage-baseline.md` if it exists.

**Empty/blank response = NOT classified — re-ask.** You MUST obtain an explicit resolution for every group.

**P3 behaviors**: Display unmapped P3 list for reference but do NOT require resolution. Record as `deferred` in coverage-baseline.md if not mapped.

### Step 2b — Post-Resolution Update

After resolving ALL unmapped P1/P2 groups:

1. **Re-run affected Features**: If any Features were marked 🔀 (re-specify) or new Features were created, execute their pipeline steps before continuing.
2. **Recalculate coverage metrics**: Include newly assigned items and new Features.
3. **Display updated metrics**:

```
📊 Updated Source Behavior Coverage:

| Priority | Total | Mapped | Excluded | Coverage |
|----------|-------|--------|----------|----------|
| P1 (core)       | [N] | [M] | [E] | [%] |
| P2 (important)  | [N] | [M] | [E] | [%] |
| P3 (nice-to-have)| [N] | [M] | [E] | [%] |
```

4. **Verify P1 + P2 = 100%** (mapped + excluded = total). If still below 100%, repeat Step 2a for remaining unmapped items.

---

## Calling Modes

### Standalone (`/smart-sdd coverage`)

- Step 1 always runs (dashboard)
- Step 2 offers "Resolve gaps now" vs "View only"
- Non-blocking — user chooses whether to act

### From Adopt Pipeline (Post-Pipeline)

- Step 1 runs
- Step 2 is **BLOCKING** — "Resolve gaps now" is mandatory
- P1 + P2 must reach 100% before adopt pipeline can complete
- "View only" is not offered

### From Rebuild Pipeline (Post-Pipeline, optional)

- Step 1 runs
- Step 2 offers "Resolve gaps now" vs "View only"
- Non-blocking — informational by default
