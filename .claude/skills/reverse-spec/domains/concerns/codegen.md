# Concern: codegen

> Generated/templated code where an IDL, schema, or template is source-of-truth.

---

## R1. Detection Signals

> See [`shared/domains/concerns/codegen.md`](../../../shared/domains/concerns/codegen.md) § Code Patterns

### Additional Detection Heuristics

- **Generated marker scan**: Files containing `@generated`, `DO NOT EDIT`, `Code generated by`, `auto-generated` in first 10 lines.
- **Repetition analysis**: Directories with 10+ files where AST/structural similarity exceeds 80%. Indicators: identical import blocks, same function signature patterns with different string constants, parallel file naming (`{entity}_service.py`, `{entity}_model.py`).
- **Generator config presence**: `buf.gen.yaml`, `openapi-generator-cli.json`, `graphql-codegen.ts`, `prisma/schema.prisma`, `sqlc.yaml`.

---

## R3. Analysis Axes — Generated Code Extraction

When codegen concern is detected, extract:

| Axis | What to Extract | Output Format |
|------|-----------------|---------------|
| Source-of-truth registry | All IDL/schema/template files that drive code generation | `{source_file, generator_tool, output_pattern}` |
| Generated file inventory | All generated output files, grouped by source | `{source → [generated_files]}` mapping |
| Manual patch zones | Generated files with manual edits (detected by mixed authorship or section markers) | `{file, manual_sections}` |
| Repetitive file groups | Directories with high structural similarity (template-driven) | `{directory, file_count, similarity%, varying_elements}` |
| Generator pipeline | Steps to regenerate all outputs from sources | Ordered command list |
