## 솔직한 이야기

spec-kit-skills를 만드는 데 3주에 걸쳐 200개 이상의 커밋이 필요했습니다. AI 에이전트를 *신뢰할 수 있게* 만드는 것이 *유능하게* 만드는 것과 근본적으로 다르다는 것을 깨달았습니다.

에이전트는 항상 Feature를 만들 수 있었습니다. 문제는 *올바른* Feature를, *올바른 방식*으로, *매번* 만들도록 하는 것이었습니다.

## 실패 패턴

### 패턴 1: 빌드 통과 ≠ 기능 동작

**함정:** 빌드 성공, 테스트 통과, 린터 클린 — 런타임에서 깨짐.

**실제 사례:** Zustand 상태 셀렉터가 매 렌더마다 새 객체 참조 생성 → 무한 리렌더링, 스트리밍 중 스크롤 깨짐. TypeScript는 문제를 못 봄.

**교훈:** 빌드 도구는 구문, 테스트는 격리된 유닛을 체크합니다. 런타임 동작은 둘 다 체크하지 않습니다. 런타임 검증이 파이프라인의 일급 단계여야 합니다.

---

### 패턴 2: 에이전트가 자기 규칙을 잊는다

**함정:** 50개 메시지 후 컨텍스트 압축 → 규칙 사라짐.

**실제 사례:** Feature 6 검증 중 verify-phases.md 참조가 압축으로 사라짐. Feature 1~5에서는 성공적으로 완료했던 Playwright 검증을 통째로 건너뜀.

**교훈:** 메타 문제입니다. 모든 안전장치는 에이전트가 지시를 읽는다고 가정합니다. 지시가 컨텍스트에서 사라지면, 아무것도 동작하지 않습니다.

**해결:** 핵심 규칙을 모든 실행 지점에 인라인. 상태를 파일로 추적.

---

### 패턴 3: 에이전트가 읽지 않고 추측한다

**함정:** 소스에서 실제 값을 읽을 수 있는데 그럴듯한 값을 생성.

**실제 사례:** CSS 색상 토큰을 소스 테마에서 읽는 대신 추측. "사이드바 탭 3개"라고 가정했는데 실제 코드에는 2개.

**교훈:** 에이전트는 생성하도록 훈련됩니다. 정답이 `grep` 한 번이면 나오는데도 추측합니다. 스킬이 소스 읽기를 *강제*해야 합니다.

---

### 패턴 4: 에러 경로는 물어보기 전에는 없다

**함정:** 스펙이 정상 경로만 기술. 에러 경로는 아무도 안 씀.

**실제 사례:** 스트리밍 채팅의 스펙이 "메시지 전송 → 응답 수신"만 정의. "로딩 → 스트리밍 → 완료 → 에러 → 정리 → 재시도"는 없음.

---

### 패턴 5: "하나만 더" 함정

**함정:** 기존 Feature에 작은 요구사항 추가 → 에이전트가 전체 스펙 재생성 → 이전 SC 전부 손실.

**해결:** `add --to` + SC Preservation. 기존 SC `[preserved]`, 새 SC `[new]`.

## 스킬 개발자를 위한 팁

### 팁 1: 스킬은 가이드가 아니라 계약이다

"구현 후 테스트 실행을 고려하세요"
→ 에이전트: "테스트는 선택사항" ❌

"차단: 테스트 실행. 실패하면 진행 금지."
→ 에이전트: "실행해야 하고 통과해야 한다" ✅

---

### 팁 2: 안티패턴이 패턴보다 중요하다

RIGHT 예시 하나는 참고점 하나. WRONG 다음에 RIGHT를 보여주면 경계가 생깁니다. 안티패턴이 다른 어떤 기법보다 디버깅 시간을 절약했습니다.

---

### 팁 3: 상태 머신이 If/Else를 이긴다

스킬에 여러 상태가 있으면 파일의 상태 머신으로 모델링하세요. `sdd-state.md`가 우리의 상태 머신입니다.

---

### 팁 4: 컨텍스트 압축을 염두에 두고 테스트하라

1. 대화 시작 → 2. 긴 워크플로우 (50+ 메시지) → 3. 에이전트가 여전히 규칙을 따르는가?

안 따르면 인라인 강제와 파일 기반 상태 추적을 강화하세요.

---

### 팁 5: 자동화 불가 = 건너뛰기가 아니라 위임

"드래그앤드롭 테스트 건너뜀" → 미테스트 기능이 조용히 통과 ❌

"파일을 업로드 영역에 드래그해주세요. 상태가 변하나요?" → 사용자가 테스트 ✅

---

### 팁 6: 스킬을 소프트웨어처럼 버전 관리하라

history.md에 40개 이상의 항목. *무엇이* 변했는지뿐 아니라 *왜* 변했는지 기록합니다.

## 더 큰 그림

**AI 에이전트가 유능해질수록, 개발자는 그 능력을 방향 짓는 더 구조화된 방법이 필요합니다.**

미래는 "AI가 모든 코드를 쓴다"가 아닙니다. 개발자가 점점 더 정교한 하네스를 만드는 것 — 도메인 전문성을 인코딩하고, 품질 게이트를 강제하며, 장기 워크플로우에서 컨텍스트를 유지하는 시스템.

오늘은 Claude Code 스킬. 내일은 다른 무언가. 하지만 패턴 — Context Continuity, Enforce Don't Reference, File over Memory — 은 계속 유효할 것입니다.

## 🤖 에이전트를 위한 글 — 스킬 개발 체크리스트

```
skill development checklist:

  structure:
    - SKILL.md 300줄 이하 (항상 로드)
    - commands/ 온디맨드 워크플로우
    - reference/ 보조 규칙

  enforcement:
    - 모든 핵심 규칙: 인라인 + 차단 게이트 + 안티패턴
    - HARD STOP: AskUserQuestion, 자동 진행 금지
    - 빈 응답: 항상 재질문

  common failures:
    - 빌드 통과 ≠ 기능 동작 → 런타임 검증 필요
    - 에이전트 규칙 망각 → 모든 실행 지점에 인라인
    - 에이전트 추측 → 소스 읽기 강제
    - 에러 경로 누락 → Brief에서 명시적으로 질문
    - 컨텍스트 손실 → 파일 기반 상태 복구

  tips:
    - 안티패턴 > 패턴
    - 상태 머신 > if/else
    - 위임 > 건너뛰기
    - 인라인 반복 30개 > 무시되는 참조 1개
```
