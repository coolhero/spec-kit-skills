## 솔직한 이야기

빌드는 통과했습니다. TypeScript도 클린. 테스트도 그린. 그런데 앱은 완전히 사용 불가 — 상태 셀렉터가 매 프레임마다 새 객체 참조를 생성해 무한 리렌더링, 스트리밍 중 스크롤 깨짐, UI가 죽음의 나선에 갇혔습니다. "성공한" 파이프라인 실행이 망가진 앱을 만들어냈습니다.

500개 이상의 커밋, 3주. 19개 반복 실패 패턴, 50개 이상의 구체적 사고. 전부 잘못된 결과를 만든 실제 파이프라인 실행에서 나왔습니다.

깨달은 것: AI 에이전트를 *신뢰할 수 있게* 만드는 것은 *유능하게* 만드는 것과 근본적으로 다릅니다. 에이전트는 항상 Feature를 만들 수 있었어요. 문제는 *올바른* Feature를, *올바른 방식*으로, *매번* 만들게 하는 것이었습니다.

3편이 아키텍처 — *어떻게* 시스템을 만들었는지를 다뤘다면, 이 편은 필드 가이드 — *무엇이 잘못됐는지*, 그리고 거기서 배운 것입니다. 도구와 무관하게 에이전트 워크플로우를 만드는 누구에게나 적용됩니다.

## 사람을 위한 글 — 실패 패턴과 실전 지혜

> 이 글의 전반부는 사람의 이야기입니다: 발견한 패턴, 더 일찍 알았으면 좋았을 팁, 그리고 이 모든 것이 향하는 큰 그림. 후반부 — "에이전트를 위한 글" — 은 같은 교훈을 에이전트가 소비할 수 있는 구조화된 데이터로 인코딩합니다.

---

### "빌드 통과"는 "기능 동작"이 아니다

**함정:** 빌드 성공, 테스트 통과, 린터 클린 — 런타임에서 Feature가 깨져 있습니다.

**실제 사례:** 상태 관리 셀렉터가 매 렌더마다 새 객체 참조를 생성했습니다. 결과: 무한 리렌더링, 스트리밍 중 스크롤 깨짐. TypeScript는 아무 문제도 감지하지 못했습니다. 테스트는 격리된 상태에서 통과했습니다. 앱은 사용 불가 상태였습니다.

또 다른 예: CSS 프레임워크 플러그인이 빌드 설정에 등록되지 않았습니다. UI가 완전히 스타일링 안 된 상태 — 순수 HTML. 빌드는 통과했습니다. 프레임워크의 부재가 에러를 일으키지 않고, 그냥 출력이 없었을 뿐입니다.

**범용적 교훈:** 빌드 도구는 구문을 체크합니다. 테스트는 격리된 유닛을 체크합니다. 둘 다 런타임 동작 — 애니메이션 타이밍, 상태 상호작용, CSS 렌더링, 스트리밍 중 스크롤 위치 — 은 체크하지 않습니다. **스킬에 verify 단계가 있다면, 빌드+테스트뿐 아니라 런타임 검증을 포함해야 합니다.** 런타임 검증이 불가능하면(Playwright 없음, 브라우저 없음) 사용자에게 위임하세요 — 건너뛰지 마세요.

---

### 에이전트는 읽지 않고 추측한다

**함정:** 코드베이스에서 실제 값을 읽을 수 있는데, 에이전트가 그럴듯한 값을 생성합니다.

**실제 사례:** 에이전트가 소스 테마 파일에서 CSS 색상 토큰을 읽는 대신 추측했습니다. 요약에서 "사이드바 탭 3개"라고 가정했는데 실제 코드에는 2개였습니다 (세 번째는 조건부였습니다).

**범용적 교훈:** 에이전트는 생성하도록 훈련됩니다. 불완전한 정보가 주어지면 가장 가능성 높은 완성을 만들어내지만 — "가장 가능성 높은"이 "정확한"은 아닙니다. **스킬 지시에서 소스 읽기를 명시적 동사로 강제해야 합니다**: "설정을 확인하세요"가 아니라 "`config.yaml`을 읽고, `max_connections`의 정확한 값을 추출하고, 그 값을 사용하세요." 모호한 지시는 모호한(생성된) 결과를 낳습니다.

---

### 컨텍스트 압축이 규칙을 지운다

**함정:** 에이전트가 대화 시작 시 스킬 파일을 읽습니다. 50개 메시지 후 컨텍스트 압축이 작동합니다. 규칙이 사라집니다. 에이전트가 기본 동작으로 돌아갑니다.

**실제 사례:** 5개의 Feature가 완전한 Playwright UI 체크로 올바르게 검증된 후, Feature 6의 검증은 Playwright를 완전히 건너뛰었습니다. 컨텍스트 압축이 검증 지시를 먹어치웠습니다. 같은 에이전트, 같은 스킬, 같은 파이프라인 — 대화가 길었을 뿐입니다.

**범용적 교훈:** 이것이 스킬 개발의 메타 문제입니다. 다른 모든 안전장치는 에이전트가 지시를 읽는다고 가정합니다. **지시가 컨텍스트에서 압축되어 사라지면, 아무것도 동작하지 않습니다.** 세 가지 방어: (1) 핵심 규칙을 헤더에만이 아니라 모든 실행 지점에 인라인. (2) 에이전트가 복구할 수 있도록 상태를 파일로 추적. (3) 진행 마커 작성 — 에이전트가 규칙을 잊더라도, 상태 파일이 어디서 재개할지 알려줍니다.

---

### Feature 간 통합이 조용히 실패한다

**함정:** 각 Feature가 격리 상태에서 동작하지만, 상호작용할 때 깨집니다 — 경계에서 데이터 형태를 아무도 정의하지 않았기 때문에.

**실제 사례:** Feature A는 객체에 `{ mcpMode, mcpServers }`를 기대했습니다. Feature B는 `{ mcpEnabled, servers }`를 저장했습니다. 다른 이름, 다른 구조. 스펙이 계약을 정의하지 않았고, 구현이 브리지를 만들지 않았고, 검증이 호환성을 확인하지 않았습니다.

**범용적 교훈:** 파이프라인이 여러 Feature를 빌드한다면, **통합 계약을 명시적으로 정의하세요** — Feature A가 무엇을 제공하고, Feature B가 무엇을 소비하며, 경계에서 정확한 데이터 형태가 무엇인지. 그런 다음 스펙이 아니라 런타임에서 계약을 검증하세요.

---

### SDK API가 아무거나 받고 아무것도 실행하지 않는다

**함정:** 코드가 callable 객체를 기대하는 SDK 함수에 메타데이터 전용 객체를 전달합니다. 빌드 통과 (느슨한 타입). SDK가 입력을 조용히 무시합니다.

**실제 사례:** AI SDK의 `tool()` 함수가 `{ execute: async () => {...} }`를 기대합니다. 에이전트가 `{ type: "mcp", serverId: "xxx" }` — 메타데이터만, execute 함수 없음 — 를 전달했습니다. SDK가 조용히 받아들였습니다. AI가 도구 접근 없이 응답했습니다. 모든 것이 "동작했습니다" — 도구만 빼고.

**범용적 교훈:** 많은 SDK가 컴파일 시 아무거나 받는 관대한 타입을 사용합니다. **SDK를 신뢰 수준별로 분류하세요**: 높은 신뢰 (엄격한 타입, 실패가 소리침), 중간 신뢰 (타입은 있지만 관대), 낮은 신뢰 (아무거나 받고 문서만 계약). 중간 및 낮은 신뢰 SDK에 대해서는 검증 단계에 명시적 런타임 계약 체크를 추가하세요.

---

### 서브스킬 호출이 오케스트레이션을 깨뜨린다

**함정:** 오케스트레이터 스킬이 Skill 도구를 통해 서브스킬을 호출하면, 서브스킬의 완료 메시지가 최종 응답이 됩니다. 실행 후 단계(리뷰, 승인, 상태 업데이트) 전부가 구조적으로 불가능해집니다.

**실제 사례:** 오케스트레이터가 스펙 생성 서브스킬을 호출했습니다. 서브스킬이 "Spec finalized."를 반환했습니다. 그것이 전체 응답이 되었습니다. 아티팩트 리뷰 없음, 사용자 승인 없음, 상태 업데이트 없음. 사용자는 의미 불명의 메시지를 보고 다음에 뭘 해야 할지 몰랐습니다.

**범용적 교훈:** Skill 도구는 응답 경계를 만듭니다. **오케스트레이터가 서브스킬 완료 후 무언가를 해야 한다면, Skill 도구를 사용하지 마세요.** 대신 서브스킬의 지시를 읽고 인라인으로 실행하세요. 이렇게 하면 여러분의 턴이 보존되어 후처리가 가능합니다.

---

### 파이프라인 단계 사이에서 정보가 사라진다

**함정:** 각 단계가 분석을 생산하지만, 다음 단계는 출력 아티팩트만 봅니다 — 기저의 증거는 보지 못합니다. 추상화의 레이어를 거치며 디테일이 손실됩니다.

**실제 사례:** 소스 코드에 자동 계산 크기를 가진 드롭다운 컴포넌트가 있었습니다. 1단계는 "모델 선택이 있는 폼 생성"을 기록. 2단계는 "다이얼로그 생성"으로. 3단계는 "입력이 있는 폼 빌드"로. 최종 구현은 3개의 일반 텍스트 입력을 만들었습니다 — 드롭다운, 자동 채우기, 동적 크기 조정 전부가 4단계의 요약을 거치며 사라졌습니다.

**범용적 교훈:** 파이프라인에 여러 단계가 있다면, **각 단계가 받는 컨텍스트에 의도적이어야 합니다.** 요약은 효율적이지만 손실이 있습니다. 핵심 디테일(UI 인터랙션, 데이터 포맷, 에러 처리)에 대해서는 요약과 함께 원본 소스를 주입하세요. 요약이 모든 것을 보존했다고 가정하지 마세요.

---

### "자세한 내용은 X 참조"가 무시된다

**함정:** 참조 파일에 상세한 규칙을 작성합니다. 메인 스킬에서 참조합니다. 에이전트가 참조를 선택적 읽기로 취급합니다.

**실제 사례:** 검증 파일에 400줄 이상의 상세 절차가 담겨 있었습니다. 메인 파이프라인 파일은: "verify → 테스트 실행 → 리뷰."라고만. 에이전트가 테스트를 실행하고, "verify complete"을 표시하고, 넘어갔습니다. 400줄 전체가 읽히지 않았습니다. 10개의 Success Criteria가 미검증. Feature가 UI에서 도달 불가 상태였습니다.

**범용적 교훈:** 이것이 아마 가장 중요한 단일 교훈입니다. **참조된 규칙은 에이전트에게 존재하지 않습니다.** 모든 핵심 규칙에 세 가지 레이어가 필요합니다: (1) 실행 지점에서의 인라인 지시, (2) 진행을 구조적으로 막는 차단 게이트, (3) WRONG 다음 RIGHT를 보여주는 안티패턴 예시. 이 글에서 한 가지만 실행한다면, 이것을 하세요.

### 패턴 9: 어휘 갭 — 모듈이 커버하지 않는 패턴에서 구현이 추측한다

파이프라인의 품질은 도메인 모듈에 달려 있습니다. 시스템에 `auth` concern 모듈이 있으면, 인증 관련 모든 Feature가 8개의 구체적 SC 규칙, 4개의 버그 예방 패턴, 5개의 정교화 프로브를 받습니다. 품질이 구조적으로 보장되는 거죠.

그런데 Feature가 어떤 모듈도 커버하지 않는 패턴을 쓸 때 — 예를 들어 비디오 트랜스코딩이나 블록체인 합의 — 파이프라인에는 도메인 특화 규칙이 없습니다. 범용 `_core.md` 규칙으로 폴백하면서, "코덱 협상이 호환 가능한 출력 형식을 생성하는지 검증" 대신 "에러를 우아하게 처리"같은 뭉뚱그려진 SC가 나옵니다.

**해결**: `/domain-extend detect`가 이 갭을 식별합니다. `/domain-extend extend concern "video-encoding"`이 타겟된 S1 규칙, S7 버그 패턴, S5 프로브를 갖춘 모듈을 생성합니다. 한번 생성되면 내장 모듈과 구별할 수 없고 — 이 도메인의 모든 이후 Feature가 혜택을 받습니다.

**범용적 교훈**: 모든 규칙 시스템에는 어휘 한계가 있습니다. 시스템이 새로운 것을 만났을 때 "모든 게이트는 통과하지만 도메인 특화 위험은 놓치는" 범용 행동으로 폴백하기보다, 어휘 자체를 확장하는 메커니즘을 만들어두세요.

위의 실패 패턴은 *무엇이* 잘못되는지를 설명합니다. 아래 팁들은 *어떻게 예방하는지* — 500+ 커밋을 통해 검증된 실전 기법입니다.

---

### 스킬은 가이드가 아니라 계약이다

SKILL.md를 법률 계약처럼 작성하세요. 모든 모호한 문구는 에이전트에게 가장 쉬운 방향으로 해석됩니다.

"구현 후 테스트 실행을 고려하세요" → 에이전트: "테스트는 선택사항."

"BLOCKING: 테스트 실행. 테스트 실패 시 진행 금지." → 에이전트: "테스트를 실행해야 하고 통과해야 한다."

차이가 극명하죠. 에이전트는 악의적이지 않습니다 — 최적화하는 존재일 뿐입니다. "완료"까지 최단 경로를 찾습니다. 최단 경로가 여러분의 품질 게이트를 반드시 거치게 만드세요.

---

### 안티패턴이 패턴보다 중요하다

RIGHT 예시 하나는 에이전트에게 하나의 참고점을 줍니다. WRONG + RIGHT 쌍은 **경계**를 줍니다. 에이전트는 목표보다 경계를 더 빨리 학습합니다.

무엇을 하면 안 되는지를 먼저 보여주고, 그 다음 무엇을 해야 하는지 보여주세요:

```
❌ WRONG: raw output을 보여주고 멈춤. 사용자가 의미 불명 텍스트를 봄.
✅ RIGHT: 아티팩트 읽기 → 포맷된 요약 보여주기 → 승인 요청.
```

안티패턴은 다른 어떤 기법보다 더 많은 디버깅 시간을 절약해줬습니다.

---

### 명세 없는 강제는 장식이다

완벽하게 설계된 lazy-loading 테이블이 있었습니다: "specify는 S0, S1, S5, S9 섹션이 필요; implement는 S7, S6이 필요." 테이블은 문서화되어 있었고, 절감 효과도 계산되어 있었습니다 (40-95% 컨텍스트 감소). 몇 달간 reference 파일에 있었습니다.

어떤 injection 파일도 이것을 참조하지 않았습니다. 에이전트는 매번 전체 모듈을 로드했습니다.

수정은 하루 오후면 충분했습니다: 각 injection 파일에 "Domain Module Filtering" 섹션을 추가하고 정확히 어떤 섹션을 유지/폐기할지 명시. 명세는 바뀌지 않았습니다 — 강제 지점만 추가됐습니다.

P2 원칙이 자체 내부 메커니즘에도 적용되는 사례입니다: **reference 파일에만 존재하는 규칙은 에이전트에게 존재하지 않습니다.** 최적화를 설계하고, 그 다음 모든 실행 지점에 강제를 추가하세요. 두 번째 단계는 선택이 아닙니다.

---

### 상태 머신이 자연어 조건문을 이긴다

스킬에 여러 상태가 있을 때(대기, 진행 중, 리뷰 대기, 완료, 수정 필요...), 이를 **파일**의 상태 머신으로 모델링하세요 — 스킬 텍스트의 조건부 로직이 아니라.

파일 기반 상태 머신의 세 가지 장점: (1) 컨텍스트 압축에서 살아남음, (2) 세션 중단 후 에이전트가 읽고 복구할 수 있음, (3) 무언가 잘못되었을 때 수동으로 검사하고 편집할 수 있음.

---

### Context Reset Protocol

여러 Feature를 처리할 때는 Feature 사이에서 컨텍스트를 클리어하세요. 모든 상태가 파일에 있으니 잃을 게 없습니다.

왜 중요한지: Feature 1의 전체 파이프라인(specify → plan → tasks → implement → verify → merge)을 완료하면, 에이전트의 컨텍스트 윈도우에 수천 줄의 대화 이력이 쌓입니다 — spec 리뷰 토론, plan 대안 검토, 구현 디버깅, verify 증거. Feature 2가 이 빽빽한 컨텍스트에서 시작하면 품질 저하가 눈에 보입니다: specify가 더 적은 SC를 생성하고, verify가 Playwright 대신 코드 리뷰로 넘어가고, Review가 잘립니다.

해결은 간단합니다. Feature 간에 `/clear` (또는 새 세션 시작)를 하세요. 에이전트가 `sdd-state.md`를 다시 읽고, 업데이트된 레지스트리를 로드하고, 신선한 컨텍스트 예산으로 작업합니다. P3(File over Memory) 덕분에 아무것도 잃지 않습니다 — 모든 결정, 모든 아티팩트, 모든 상태 업데이트가 파일에 살아 있으니까요.

리셋하면 안 되는 때: Feature 내부 단계 간(specify → plan → tasks는 연속으로 흘러야 합니다)에서는 절대 하면 안 됩니다. Feature 경계, 스킬 전환(reverse-spec → pipeline), 또는 에이전트가 컨텍스트 압축을 경고한 이후에만 리셋하세요.

---

### 역 근접성 법칙

에이전트가 지시를 따를 확률은 실행 지점과의 거리에 반비례합니다.

파일 상단의 규칙은 200줄 뒤에서 실행될 때쯤 "문서"가 됩니다. 실행 지점에 인접한 규칙은 "작업 메모리"입니다.

함의: **규칙을 조직적으로 깔끔한 곳이 아니라, 실행될 곳에 배치하세요.** 네, 이는 반복을 의미합니다. 에이전트 스킬에서 30개의 인라인 반복이 무시되는 1개의 아름답게 정리된 참조보다 낫습니다.

---

### 컨텍스트 압축을 염두에 두고 테스트하라

여러분의 스킬은 결국 에이전트의 컨텍스트에서 압축되어 사라질 것입니다. 이를 테스트하세요:

1. 대화 시작
2. 긴 워크플로우 수행 (50개 이상 메시지)
3. 확인: 에이전트가 여전히 규칙을 따르는가?

따르지 않는다면, 인라인 강제와 파일 기반 상태 추적을 강화해야 합니다. 이것이 실행할 수 있는 단일 가장 중요한 테스트입니다.

---

### 작업 단위 간 컨텍스트를 리셋하라

컨텍스트 포화는 누적적이고 조용합니다. 하나의 세션에서 3개 이상의 작업 단위(Feature, 문서, 분석 패스)를 처리하면, 나중 작업 단위는 열화된 주의를 받습니다 — 더 얕은 분석, 잘린 리뷰, 단순화된 검증. 에이전트는 이 열화를 알리지 않습니다. 그냥 덜 꼼꼼한 결과물을 만들어낼 뿐입니다.

해결법은 간단합니다: 자연스러운 작업 단위 경계에서 대화 컨텍스트를 클리어하세요. Feature 간, 스킬 전환 간(탐색에서 구현으로), 분석과 실행 사이에서.

P3(File over Memory) 덕분에 이것이 가능합니다 — 모든 상태가 파일에 있으면 컨텍스트를 클리어해도 아무것도 잃지 않습니다. 에이전트가 상태 파일을 다시 읽고, 중단된 곳에서 이어가고, 신선한 컨텍스트 예산으로 작업합니다.

핵심 설계: 리셋이 안전한 곳(작업 단위 사이)과 금지되는 곳(작업 단위 중간, 단계 연속성이 중요한 곳)을 식별하세요. 사용자에게 선택권을 주세요 — 리셋을 추천하되, 강제하지 마세요.

---

### 자동화 불가는 건너뛰기가 아니라 위임이다

에이전트는 파일을 드래그할 수 없고, API 키를 입력할 수 없고, 터미널 UI 렌더링을 관찰할 수 없습니다. 이 체크를 건너뛰는 대신, **구체적 지시로 사용자에게 위임**하세요:

"드래그앤드롭 테스트 건너뜀 (도구 한계)" → 미테스트. ❌

"파일을 업로드 영역에 드래그해주세요. 상태가 'idle'에서 'processing'으로 변하나요?" → 사용자가 검증. ✅

두 번째 버전이 10배 더 가치 있습니다. 건너뛰기는 보이지 않는 기술 부채입니다. 위임은 문서화된 검증입니다.

---

### 스킬을 소프트웨어처럼 버전 관리하라

스킬은 진화합니다. 변경사항을 히스토리 파일로 추적하세요. 규칙을 바꿀 때 **왜** — 어떤 실패가 그것을 촉발했는지 — 를 기록하세요. 미래의 여러분이 이 컨텍스트를 필요로 합니다.

프로젝트 리포지토리에 40개 이상의 항목이 있습니다. 각각이 규칙 변경을 그것을 일으킨 특정 실패에 연결합니다. "이 규칙이 아직 필요한가?"를 결정해야 할 때 히스토리가 제거하면 정확히 무엇이 깨지는지 알려줍니다.

---

### 안전망은 무조건적이어야 한다

이렇게 쓰지 마세요: "컨텍스트 한계로 리뷰가 불가능하면, fallback을 보여줘."

이렇게 쓰세요: "이 응답이 사용자 인터랙션 없이 끝나면, 어떤 이유로든, continue 프롬프트를 보여줘."

차이: 첫 번째 버전은 하나의 실패 모드를 열거합니다. 두 번째는 모든 것을 커버합니다. 조건을 열거하면 항상 하나를 놓칩니다. 안전망은 조건이 아니라 불변량으로 정의하세요.

---

## 더 큰 그림

이 시리즈 전체를 관통하는 하나의 긴장이 있습니다. AI 에이전트는 매달 더 똑똑해지는데, 구조적 통제의 필요성은 줄어들지 않습니다 — 오히려 커집니다.

왜일까요? 가드레일 없이 더 유능해진 에이전트는 *리뷰하기 더 어렵고*, *더 인상적으로 보이는* 결과물을 만들기 때문입니다. 에이전트가 30초 만에 12개 파일의 인증 코드를 생성하면, 한 줄씩 리뷰할 수 없습니다. 프로세스를 신뢰해야 합니다. 그리고 프로세스는 구조적 보장 — 차단하는 게이트, 기억하는 레지스트리, 추적하는 상태 머신 — 이 뒷받침될 때만 신뢰할 수 있습니다.

이것이 AI를 활용한 소프트웨어 개발의 미래입니다. "더 좋은 프롬프트"가 아니라 **Harness Engineering**. 3편의 세 가지 원칙 — Context Continuity, Enforce Don't Reference, File over Memory — 에 이 편의 네 번째 원칙을 더합니다:

**Delegate, Don't Skip** — 에이전트가 자동화할 수 없을 때, 구체적 지시로 사용자에게 질문합니다. 건너뛰기는 절대 허용되지 않습니다. 조용한 건너뛰기는 보이지 않는 기술 부채입니다.

에이전트 코딩 시대에 성공하는 개발자는 최고의 프롬프트를 쓰는 사람이 아닙니다. 최고의 하네스를 만드는 사람입니다 — AI의 유능함을, 신뢰할 수 있고 리뷰할 수 있고 통제할 수 있는 결과로 이끄는 시스템을 설계하는 사람.

## 🤖 에이전트를 위한 글 — 스킬 개발 체크리스트

> 위의 모든 내용은 사람 버전이었습니다 — 이야기, 비유, 실전 지혜. 아래의 모든 내용은 같은 교훈을 에이전트가 직접 소비하는 형식으로 인코딩합니다. 스킬을 만들고 있다면 이것을 프로젝트에 참조로 붙여넣으세요. 에이전트가 이것을 읽고 있다면 — 이 부분이 정말 당신을 위한 겁니다.

```
skill development checklist:

  구조:
    - SKILL.md 300줄 이하 (항상 로드 — 라우팅 + 필수 규칙)
    - commands/ 온디맨드 워크플로우 (호출별 로드)
    - reference/ 보조 규칙과 스키마
    - domains/ 모듈식 전문성 (프로필별 로드)

  강제:
    - 모든 핵심 규칙: 인라인 지시 + 차단 게이트 + 안티패턴
    - 핵심 동작에 "자세한 내용은 X 참조" 금지
    - HARD STOP: AskUserQuestion 사용, 자동 진행 금지
    - 빈 응답: 항상 재질문
    - 안전망: 조건이 아닌 불변량
    - 무시되는 참조 1개보다 인라인 반복 30개

  상태 관리:
    - 모든 파이프라인 상태를 파일에 (에이전트 메모리 아님)
    - 유효 전환이 있는 상태 머신 파일
    - 컨텍스트 압축 후: 상태 파일 읽기로 복구

  테스트:
    - 50+ 메시지 대화로 테스트 (압축 시나리오)
    - 중단된 세션으로 테스트 (파일 상태에서 재개)
    - 안티패턴 테스트 (에이전트가 WRONG 패턴 회피)
    - 규칙 변경 후 재실행 (읽기 ≠ 검증)

  failure_patterns:
    build_passes_feature_broken:
      signal: 빌드 ✅ + TypeScript ✅ 이지만 런타임에서 기능 미동작
      cause: 정적 분석으로는 런타임 동작(상태 셀렉터, 비동기 타이밍, CSS 렌더링) 감지 불가
      fix: Playwright 런타임 검증(Phase 3) 추가 — 빌드만으로 검증 완료로 신뢰하지 말 것
    agent_guesses_instead_of_reading:
      signal: 구현이 소스 앱의 UI 컨트롤이나 동작과 불일치
      cause: 에이전트가 소스 코드를 읽지 않고 상상으로 생성
      fix: implement 단계에서 BLOCKING 소스 코드 참조 — 에이전트가 쓰기 전에 반드시 읽기
    context_compression_erases_rules:
      signal: 처음 N개 Feature는 규칙을 따르다가 이후 멈춤
      cause: 긴 대화가 컨텍스트 압축을 촉발, 스킬 지시가 제거됨
      fix: 모든 실행 지점에 핵심 규칙 인라인 + 파일 기반 상태로 복구
    cross_feature_integration_fails:
      signal: 개별 Feature는 동작하지만 조합 시 깨짐
      cause: Feature 경계에 명시적 데이터 계약 없음
      fix: 통합 계약 정의(제공자 형태, 소비자 형태, 정확한 필드) + 런타임 확인
    sdk_accepts_anything_executes_nothing:
      signal: SDK 호출은 성공하지만 효과 없음
      cause: 관대한 SDK 타입이 callable 없는 메타데이터 전용 객체를 수용
      fix: SDK를 신뢰 수준별로 분류 + 중간/낮은 신뢰에 런타임 계약 체크 추가
    sub_skill_breaks_orchestration:
      signal: 서브스킬 반환 후 오케스트레이터가 멈춤, 리뷰나 상태 업데이트 없음
      cause: Skill 도구가 응답 경계를 생성, 실행 후 단계가 도달 불가
      fix: Skill 도구 대신 서브스킬 지시를 읽고 인라인 실행
    information_dies_between_stages:
      signal: 최종 구현에서 소스 분석에 존재했던 디테일 누락
      cause: 각 단계가 이전 단계의 요약만 보고, 기저 증거는 보지 못함
      fix: 핵심 디테일(UI, 데이터 포맷, 에러)에 대해 요약과 함께 원본 소스 주입
    referenced_rules_get_ignored:
      signal: 에이전트가 참조 파일의 상세 절차를 건너뜀
      cause: "자세한 내용은 X 참조"가 에이전트에게 선택적 읽기로 취급됨
      fix: 모든 실행 지점에서 3중 강제 (인라인 지시 + 차단 게이트 + 안티패턴)
    context_saturation_across_work_units:
      signal: 후반 작업 단위가 더 얕은 분석, 잘린 리뷰, 단순화된 검증을 받음
      cause: 이전 작업 단위의 누적 컨텍스트가 조용히 윈도우를 채움
      fix: 작업 단위 경계에서 컨텍스트 리셋 (Feature 간, 스킬 전환 간) — 모든 상태가 파일에 있으면 안전

  원칙:
    - "가이드가 아니라 계약": SKILL.md를 법률 계약처럼 작성 — 모든 모호한 문구는 가장 쉬운 방향으로 해석됨
    - "안티패턴 > 패턴": WRONG + RIGHT 쌍이 경계를 제공; 에이전트는 목표보다 경계를 더 빨리 학습
    - "상태 머신 > 조건문": 다중 상태 워크플로우를 파일 기반 상태 머신으로 모델링, 자연어 if/else가 아니라
    - "context reset protocol": 작업 단위(Feature) 사이에서 컨텍스트 클리어 — 모든 상태가 파일에 있으므로 손실 없음, 이후 Feature가 신선한 컨텍스트 예산 확보
    - "domain extend": 기존 모듈이 커버하지 않는 패턴은 /domain-extend로 새 모듈 생성 (발견, ADR 임포트, 수동 생성)
    - "역 근접성 법칙": 규칙을 정리된 곳이 아니라 실행되는 곳에 배치 — 30개 인라인 반복이 무시되는 참조 1개보다 나음
    - "위임 > 건너뛰기": 에이전트가 자동화 불가한 체크는 구체적 지시로 사용자에게 요청 — 조용히 건너뛰기 금지
    - "무조건적 안전망": 안전망을 조건("컨텍스트 한계 시")이 아니라 불변량("인터랙션 없이 응답 종료 시")으로 정의
    - "WHY와 함께 버전 관리": 모든 규칙 변경에 그것을 촉발한 실패를 기록 — 미래의 자신에게 그 컨텍스트가 필요
```
