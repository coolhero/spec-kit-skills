## 왜 400개의 마크다운 파일인가?

Cursor, Windsurf, Claude Code — 요즘의 에이전트 코딩 도구들은 놀라울 정도로 유능합니다. 파일 간 설계를 하고, 컨텍스트를 보고 리팩토링하고, 테스트 결과를 기반으로 반복합니다. 그런데 왜 3주를 들여 400개의 마크다운 파일로 에이전트를 "교육"해야 할까요?

솔직한 대답: **유능한 것과 통제 가능한 것은 다르기 때문입니다.**

유능한 에이전트는 30초 만에 인증 시스템을 만들 수 있습니다. 통제 가능한 에이전트는 *팀이 결정한 방식대로* — 세션 기반으로(JWT 아닌), 서버 사이드에 토큰을 저장하고(localStorage 아닌), Feature 1이 이미 정의한 User 모델을 참조해서(새로 만들지 않고) 만듭니다. 이 둘의 차이가 전부입니다.

그리고 가장 놀라웠던 부분: **에이전트가 똑똑할수록 통제하기가 더 어렵습니다.** 한 줄짜리 자동완성은 리뷰가 쉽습니다. 12개 파일에 걸쳐 인증, 에러 처리, DB 마이그레이션을 스캐폴딩하는 에이전트는? 한 줄씩 리뷰할 수 없습니다. 코드를 생성한 *프로세스*를 신뢰해야 합니다.

이 글은 그 프로세스에 관한 것입니다. 스킬 자체가 아니라(2편에서 다뤘으니), 400개 파일이 하나의 일관된 시스템으로 작동하게 하는 **아키텍처 원칙** — 그리고 에이전트 워크플로우를 만드는 누구에게나 이 원칙이 왜 중요한지.

## AI 스킬 설계의 역설

AI 에이전트를 위한 도구를 만들 때의 이상한 진실: **에이전트가 사용자이자 런타임입니다.**

React 컴포넌트를 작성하면 React가 실행합니다. Claude Code 스킬을 작성하면 Claude가 마크다운으로 *읽고* 어떻게 행동할지 결정합니다. "코드"는 자연어. "컴파일러"는 LLM. "버그"는 행동적입니다 — 구문 오류 때문이 아니라 의도를 오해해서 에이전트가 잘못된 일을 합니다.

이것은 스택 트레이스로 디버깅할 수 없다는 뜻입니다. 브레이크포인트를 설정할 수 없습니다. "에이전트가 X 지시를 따르는가"에 대한 유닛 테스트를 작성할 수 없습니다. 유일한 테스트 방법은: 파이프라인을 실행하고, 무슨 일이 일어나는지 관찰하고, 반복하는 것입니다.

거기에 더해 이것을 특별히 어렵게 만드는 추가 제약이 있습니다: **컨텍스트 윈도우는 유한하며 압축됩니다.** 스킬 파일 상단에 정성껏 쓴 규칙이? 50개 메시지 후에는 한 문장 요약으로 압축되거나 — 아예 삭제될 수 있습니다.

실제로 겪은 사례를 하나 들겠습니다. 검증(verify) 단계에 "머지 전에 항상 전체 검증을 실행하라"는 규칙을 명시했습니다. Feature 1부터 5까지는 에이전트가 4개 Phase를 빠짐없이 수행했습니다. 그런데 Feature 6에서 — 대화가 50개 메시지를 넘긴 시점에서 — 에이전트가 갑자기 `build + TypeScript check`만 실행하고 "verify 완료"를 선언했습니다. 나머지 Phase는 전부 건너뛰었습니다. 무슨 일이 벌어진 걸까요? 컨텍스트 압축이 상세 규칙을 "verify = run build"로 요약해버린 겁니다. 4단계짜리 프로토콜이 한 줄로 축소된 거죠. 이건 에이전트의 잘못이 아닙니다 — 압축된 컨텍스트에서는 그게 전부였으니까요.

이것이 설계 방식 전체를 바꿉니다. "좋은 지시를 쓰는 것"만으로는 충분하지 않습니다. 지시가 압축되고, 삭제되고, 재해석되는 환경에서 여전히 동작하도록 **시스템을 설계**해야 합니다.

일반 소프트웨어에서의 비유를 들자면: 함수 시그니처가 실행 중에 랜덤하게 변하고, import문이 가끔 무시되고, 주석이 어느 순간 코드로 실행되는 환경을 상상해보세요. 이런 환경에서 안정적인 시스템을 만들려면 "좋은 코드를 쓰는 것" 이상의 무언가가 필요합니다. 방어적 설계, 중복된 안전장치, 그리고 "이 규칙이 사라져도 시스템이 여전히 올바르게 동작하는가?"라는 질문을 모든 결정에 적용해야 합니다.

이 관점에서 보면, 에이전트 스킬 설계는 분산 시스템 설계와 비슷합니다. 분산 시스템에서는 네트워크 파티션을 가정하고 설계합니다 — "이 노드가 다른 노드와 통신할 수 없을 때도 시스템이 동작해야 한다." 에이전트 스킬에서는 컨텍스트 파티션을 가정합니다 — "이 규칙이 에이전트의 활성 컨텍스트에서 사라졌을 때도 파이프라인이 올바르게 동작해야 한다."

분산 시스템에 eventual consistency가 있듯, 에이전트 스킬에는 "file-based consistency"가 있습니다 — 에이전트의 메모리가 아무리 불안정해도, 파일에 기록된 상태는 항상 일관적입니다. 분산 시스템에 idempotent operations가 있듯, 에이전트 스킬에는 "재시작 안전성(restart safety)"이 있습니다 — 에이전트가 어느 시점에서 중단되고 새 세션에서 재시작해도, 파일 상태만 읽으면 정확히 이어갈 수 있습니다.

이 비유를 더 확장하면: 분산 시스템에서 "모든 노드가 동시에 죽을 수 있다"는 최악의 시나리오를 대비하듯, 에이전트 스킬에서는 "모든 컨텍스트가 한 번에 압축될 수 있다"는 시나리오를 대비합니다. 그 대비가 바로 P3의 핵심입니다 — 파일에 있으면 살아남고, 메모리에만 있으면 사라집니다.

실제로 이 "최악의 시나리오"는 드물지 않습니다. 긴 파이프라인 실행에서 사용자가 중간에 디버깅 대화를 하거나, 에이전트에게 코드베이스의 다른 부분을 탐색시키면, 컨텍스트 윈도우가 빠르게 소모됩니다. 그 결과로 초기 파이프라인 컨텍스트가 대폭 압축되는 것은 흔한 일입니다.

## 세 가지 기초 철학

모든 설계 결정은 세 가지 원칙으로 거슬러 올라갑니다. 처음부터 있었던 게 아닙니다 — 수백 번의 실패한 파이프라인 실행을 통해 발견했습니다.

세 원칙의 관계를 간략히 설명하면:

- **P1 (Context Continuity)**: *무엇을* 보존해야 하는가 — Domain Profile, 소스 코드 분석, Feature 간 정보
- **P2 (Enforce, Don't Reference)**: *어떻게* 보존하는가 — 참조가 아닌 인라인 강제, 차단 게이트, 안티패턴 예시
- **P3 (File over Memory)**: *어디에* 보존하는가 — 에이전트 메모리가 아닌 파일 시스템

세 원칙은 독립적이 아니라 상호 보완적입니다. P1이 "이 정보가 필요하다"고 말하면, P3가 "파일에 저장하라"고 말하고, P2가 "그 파일을 읽도록 강제하라"고 말합니다. 하나라도 빠지면 시스템이 무너집니다:

- P1 없이 P2+P3: 무엇을 저장하고 강제할지 모릅니다. 임의의 규칙이 강제될 뿐.
- P2 없이 P1+P3: 정보가 파일에 있지만, 에이전트가 읽지 않습니다. "아름다운 문서가 무시되는" 상태.
- P3 없이 P1+P2: 규칙이 강제되지만, 세션이 바뀌면 에이전트가 모든 것을 잊습니다.

### P1: Context Continuity — 정보는 앞으로 흘러야 한다

> 정보는 모든 파이프라인 단계를 통해 연속적으로 흘러야 한다. 전환에서 아무것도 손실되면 안 된다.

당연해 보이지만, AI 에이전트에서 "전환"이 무엇을 의미하는지 깨닫기 전까지는 그렇습니다:

**Feature 간:** Feature 3은 깨끗한 컨텍스트에서 시작합니다. Feature 1의 User 엔티티나 Feature 2의 API 계약에 대해 아무것도 모릅니다 — 명시적으로 그 정보를 로드하지 않으면. 이를 Global Evolution Layer로 해결했습니다: entity-registry.md, api-registry.md, sdd-state.md가 정보를 파일로 앞으로 전달합니다.

**파이프라인 단계 간:** `speckit-specify`가 끝나고 `speckit-plan`이 시작될 때, 에이전트의 작업 컨텍스트가 전환됩니다. specify의 상세 분석이 압축될 수 있습니다. 이를 각 단계별로 관련 컨텍스트를 명시적으로 재로드하는 injection 파일로 해결했습니다.

**세션 간:** 사용자가 노트북을 닫고 다음 날 열어서 계속합니다. 에이전트가 어제 "알고 있던" 모든 것이 사라집니다 — 파일에 있는 것만 빼고. 모든 중간 결과, 모든 결정, 모든 상태 전환이 파일 시스템에 영속화됩니다.

이 세 가지 전환이 위험한 이유는, 각각에서 **정보 손실이 조용하게** 일어나기 때문입니다. 에이전트가 "이 정보를 잊었습니다"라고 알려주지 않습니다. 자신감 있게 잘못된 가정을 기반으로 진행합니다. Feature 1에서 User 엔티티에 UUID를 사용하기로 결정했는데, Feature 3에서 에이전트가 자동 증가 정수를 사용하는 User를 생성합니다 — 아무런 경고 없이요. 사실 에이전트 입장에서는 경고할 이유가 없습니다. Feature 1의 결정이 컨텍스트에 없으니, 자동 증가 정수가 합리적인 선택이니까요.

이것이 일반 소프트웨어의 버그와 다른 점입니다. 일반 버그는 "무언가가 잘못되었다"는 에러를 발생시킵니다. 컨텍스트 손실로 인한 문제는 에러 없이 조용히 잘못된 결과를 만들어냅니다. 발견하기 어렵고, 발견한 후에도 원인을 추적하기 어렵습니다.

Context Continuity에는 세 가지 하위 원칙이 있습니다:

**P1-a: Domain Profile은 First-Class Citizen.** 일회성 설정이 아닙니다. Domain Profile이 모든 단계를 적극적으로 형성합니다 — `add`에서 어떤 프로브가 질문되는지, `specify`에서 어떤 규칙이 활성화되는지, `plan`에서 어떤 아키텍처 결정이 필수인지, `implement`에서 어떤 패턴을 따라야 하는지, `verify`에서 어떤 검증 단계가 실행되는지. 프로필이 `gui + realtime`이면, specify는 낙관적 UI 업데이트와 재연결 처리를 위한 SC를 생성합니다. `cli + resilience`면, 완전히 다른 SC가 나옵니다 — 재시도 전략, 타임아웃 처리, 부분 실패 복구에 초점을 맞춥니다.

구체적인 메커니즘은 이렇습니다: 도메인 모듈에는 표준화된 섹션이 있습니다 (S1은 SC 생성 규칙, S5는 정교화 프로브, S7은 버그 방지 규칙, S8은 런타임 검증 전략). 파이프라인이 실행될 때, injection 파일이 관련 모듈을 로드하고 해당 섹션을 병합합니다. `ai-assistant` 아키타입은 A2 (토큰 관리, 스트리밍 중단을 위한 SC 확장)를 추가합니다. `gui` 인터페이스는 S6 (UI 테스트 통합)을 추가합니다. 누적됩니다 — specify의 최종 컨텍스트는 모든 활성 모듈의 S1 섹션의 합집합입니다.

여기서 정말 흥미로운 건 **Cross-Concern Integration Rules**입니다 — Concern 조합에서 발생하는 창발적 규칙입니다. 단일 Concern만으로는 드러나지 않는 문제가 있습니다. 현재 시스템에는 46개의 조합 패턴이 정의되어 있습니다.

예를 들어 `gui + realtime` 조합을 봅시다. `gui`만 활성화되면 표준적인 UI 규칙이 나옵니다 — 폼 검증, 내비게이션 가드, 호버 상태. `realtime`만 활성화되면 스트리밍 프로토콜 규칙이 나옵니다 — 연결 관리, 메시지 순서 보장, 재연결 로직. 그런데 둘이 **함께** 활성화되면? 어느 쪽에서도 나오지 않는 새로운 규칙이 필요합니다:

- **Optimistic Update**: 서버 응답 전에 UI를 먼저 업데이트하되, 실패 시 롤백
- **Reconnection UI**: WebSocket이 끊겼을 때 사용자에게 보여줄 상태 표시
- **"Stale UI after Reconnect" Prevention**: 재연결 후 화면에 남아있는 오래된 데이터를 감지하고 갱신

이 세 번째가 특히 교활합니다. WebSocket이 30초간 끊겼다 다시 연결된 후, 화면의 채팅 메시지 목록이 30초 전 상태 그대로입니다. 사용자는 연결이 복구되었으니 최신 상태라고 생각하지만, 실제로는 그 30초 동안의 메시지가 누락되어 있습니다. `gui` 모듈도, `realtime` 모듈도 이 시나리오를 단독으로는 다루지 않습니다 — 둘의 교차점에서만 발생하니까요.

다른 조합도 마찬가지입니다. `auth + realtime`은 "토큰 만료 중 스트리밍"이라는 창발적 시나리오를 만듭니다 — 스트리밍 도중에 인증 토큰이 만료되면 어떻게 해야 할까요? 스트림을 끊고 재인증? 백그라운드에서 토큰을 갱신하면서 스트림 유지? `auth` 모듈과 `realtime` 모듈 각각은 이 시나리오를 다루지 않지만, 조합 패턴이 이를 명시적으로 처리합니다. 이런 식으로 46개의 조합이 실제 프로젝트에서 발견된 교차점 문제를 사전에 방지합니다.

**P1-b: Artifact Separation (소스 코드 충실성).** 이 원칙은 고통스러운 실패에서 태어났습니다. 기존 앱을 더 나은 아키텍처로 다시 만드는 "리빌드" 프로젝트였습니다. reverse-spec으로 기존 앱을 분석하고, smart-sdd로 새로 구현하는 흐름이었죠. 그런데 Feature 3의 스펙이 원하는 동작 대신 소스 앱의 구현 디테일을 기술했습니다. 다르게 구현하려고 하니(더 나은 데이터 모델, 다른 API 설계) 스펙이 저항했습니다 — 옛 구현 쪽으로 계속 끌려갔습니다.

구체적인 사례를 하나 더 들어보겠습니다. Knowledge Base 생성 UI에 ModelSelector 드롭다운이 있었습니다. 사용자가 모델을 선택하면 임베딩 차원이 자동으로 계산되는 구조였습니다. reverse-spec의 SBI(Source Behavior Inventory)는 이를 정확히 기록했습니다: "KB를 이름, 모델 선택, 자동 계산된 차원으로 생성." 그런데 specify → plan → implement를 거치면서, 소스의 "모델 선택 드롭다운 + 자동 차원 계산"이 "이름, 모델, 차원 — 세 개의 일반 텍스트 입력"으로 퇴화했습니다. 모델 선택이라는 핵심 인터랙션이 사라지고, 사용자가 차원 수를 직접 입력하는 구현이 나온 겁니다.

왜 이런 일이 벌어졌을까요? 스펙이 "무엇을 만들 것인가"에만 집중하면서, 소스 앱의 구현 패턴을 참조할 통로가 없었기 때문입니다. Artifact Separation은 맞지만, 소스의 행동적 본질(behavioral essence)까지 잃으면 안 됩니다.

해결책은 **Source Code Reference Injection**을 BLOCKING gate로 만드는 것이었습니다. implement 단계에 진입할 때, injection 파일이 pre-context.md에서 해당 Feature의 소스 행동 요약을 자동으로 로드합니다. "KB 생성: ModelSelector → 차원 자동 계산"이라는 핵심 패턴이 implement 컨텍스트에 존재하므로, 에이전트가 이를 일반 텍스트 입력으로 단순화하려 하면 pre-context와의 충돌을 감지할 수 있습니다.

이것은 미묘한 균형입니다.

스펙은 소스에 독립적이어야 합니다 — 어떻게 만들지의 자유.
구현은 소스의 행동적 본질을 보존해야 합니다 — 사용자 경험의 연속성.

이 두 가지 요구를 동시에 만족시키려면, 정보가 파이프라인의 **다른 지점**에서 다른 방식으로 제공되어야 합니다. specify 시점에는 소스 정보를 차단하고(창의적 자유를 위해), implement 시점에는 소스 정보를 주입합니다(행동적 충실성을 위해). 같은 정보가 파이프라인 단계에 따라 "독"이 될 수도, "약"이 될 수도 있다는 것이 핵심 통찰입니다.

스펙은 "무엇을 만들 것인가"만 기술하고, "어디서 왔는가"는 절대 기술하지 않습니다. 소스 분석은 reverse-spec 아티팩트(pre-context.md)에 남습니다. Smart-sdd 스펙(spec.md)은 소스에 무관합니다. 이 분리 덕분에 스펙이 옛 구현에 오염되지 않은 채로 다르게 재구축할 수 있습니다. 하지만 implement 시점에는 pre-context를 통해 소스의 행동적 본질을 다시 참조합니다.

**P1-c: Cross-Feature Memory (GEL).** Global Evolution Layer는 Feature 간 정보를 전달하는 파일로 구성됩니다:
- `entity-registry.md` — 필드, 타입, 관계를 가진 모든 데이터 모델
- `api-registry.md` — 계약을 가진 모든 엔드포인트
- `sdd-state.md` — 프로젝트 상태 머신 (어떤 Feature가 어떤 단계에 있는지)
- `roadmap.md` — Feature 의존성 그래프
- `pre-context.md` (Feature별) — 상세 요구사항 컨텍스트
- `stubs.md` (선택) — Feature 간 인터페이스 스텁 (아직 구현되지 않은 Feature의 API 시그니처)

Feature 3이 시작될 때, 에이전트가 레지스트리를 읽습니다. Feature 1의 User 엔티티를 보고, 가상의 필드가 아니라 기존 필드를 참조하는 SC를 생성합니다. 돌이켜보면 너무나 당연한데 — 이것 없이는 모든 Feature가 모든 엔티티를 재발명하고, 통합이 악몽이 됩니다.

GEL이 없으면 정확히 무슨 일이 벌어지는지 하나 보겠습니다. 초기 버전에서 GEL 없이 Feature 3을 진행했을 때, 에이전트가 자체적으로 User 엔티티를 정의했습니다 — Feature 1에서 이미 정의된 것과 다른 필드명, 다른 타입으로요. Feature 1의 User에는 `displayName: string`이 있었고, Feature 3의 User에는 `username: string`과 `fullName: string`이 있었습니다. API도 마찬가지였습니다 — Feature 2가 `POST /api/documents`를 정의했는데, Feature 3이 같은 엔드포인트를 `POST /api/files`로 재정의했습니다. 두 Feature가 통합되는 시점에서 필드 충돌, API 충돌, 타입 불일치가 연쇄적으로 터졌습니다.

이 문제가 특히 위험한 이유는, 각 Feature를 개별적으로 보면 **완벽하게 동작한다**는 점입니다. Feature 3의 코드는 Feature 3의 스펙에 정확히 맞습니다. 빌드도 통과하고, 타입 체크도 통과합니다. 문제는 Feature 1과 Feature 3을 **같은 코드베이스에 합칠 때** 비로소 드러납니다. 이 시점에서는 이미 두 Feature의 구현이 끝난 상태이므로, 수정 비용이 엄청납니다. 하나의 Feature에서 엔티티 스키마를 바꾸면, 그 엔티티를 참조하는 모든 Feature의 코드를 수정해야 합니다. 연쇄적 수정이 새로운 버그를 만들고, 그 버그를 고치면 또 다른 Feature에 영향을 미칩니다.

GEL의 레지스트리는 이 문제를 구조적으로 방지합니다 — Feature 3의 에이전트가 기존 User 엔티티를 **보고** 시작하니까요. 충돌은 구현 시점이 아니라 specify 시점에 감지됩니다.

**Context Reset Protocol — 작업 단위 간 P1 운영화**

File over Memory의 반직관적 함의가 있습니다: 모든 것이 파일에 있다면, 주기적으로 대화 컨텍스트를 *버려야* 합니다. Feature 하나를 완료(main 머지)한 후, 에이전트의 컨텍스트 윈도우는 그 Feature의 대화 이력으로 가득합니다 — spec 리뷰, plan 논의, 구현 세부사항, verify 결과. 그 짐을 안고 다음 Feature를 시작하면 새 Feature의 injection과 review를 위한 공간이 줄어듭니다.

Context Reset Protocol: 작업 단위 간에 컨텍스트를 비우세요. `/clear` 후 다시 호출. 에이전트가 sdd-state.md를 다시 읽고, 업데이트된 레지스트리를 로드하고, 새로 시작합니다. 모든 것이 파일에 기록되었으므로 아무것도 잃지 않습니다. 얻는 것은 다음 Feature를 위한 전체 컨텍스트 예산입니다.

리셋하면 안 되는 때: Feature 내부(specify와 plan 사이)에서는 절대. 단계 간 연속성이 끊기지 않는 흐름을 요구합니다. Feature 경계, 스킬 전환, 또는 컨텍스트 압축 경고 이후에만 리셋하세요. *(실전 구현 — 언제 리셋하고, 무엇이 보존되고, 어떻게 테스트하는지는 4편 "Context Reset Protocol" 섹션을 참조하세요.)*

**네 번째 스킬의 아키텍처적 역할**

세 파이프라인 스킬(code-explore, reverse-spec, smart-sdd)은 도메인 모듈을 소비합니다. 그런데 누가 모듈을 만들까요? 처음에는 프레임워크가 47개 concern, 15개 archetype, 40개 이상의 foundation과 함께 출시됩니다. 탄탄한 시작 어휘이지만, 모든 프로젝트는 결국 커버되지 않는 패턴을 만납니다.

`/domain-extend`가 이 루프를 닫습니다. 파이프라인 단계가 아니라 파이프라인이 사용하는 어휘를 풍부하게 하는 **메타 도구**입니다. code-explore가 알 수 없는 패턴을 감지하면, domain-extend가 그것을 위한 모듈을 만들 수 있습니다. 팀이 ADR과 포스트모템을 임포트하면, domain-extend가 그 조직적 지식을 구조화된 규칙으로 변환합니다. 결과: spec-kit-skills를 사용하는 각 프로젝트가 다음 프로젝트를 위해 시스템을 더 똑똑하게 만듭니다.

이것은 P1(Context Continuity)을 Feature 간이 아닌 프로젝트 간에 적용한 것입니다.

---

### P2: Enforce, Don't Reference — 가장 반직관적인 원칙

> "자세한 내용은 X 참조"는 강제력이 없다. 규칙은 실행 지점에서 직접 강제되어야지, 먼 곳에서 참조되면 안 된다.

일반 소프트웨어에서는 함수를 한 번 작성하고 여러 곳에서 호출합니다. DRY — Don't Repeat Yourself. 하지만 에이전트 스킬에서는 **DRY가 컴플라이언스를 죽입니다.**

이유는 이렇습니다: 에이전트는 참조를 선택적 읽기로 취급합니다. "전체 검증 프로토콜은 verify-phases.md 참조"를 보면, "더 많은 정보가 있지만, 진행하기에 충분한 컨텍스트가 있다"로 처리합니다. 빌드 + TypeScript를 실행하고, "verify ✅"를 보고하고, 넘어갑니다. verify-phases.md의 400줄 상세 검증 로직은 읽히지 않습니다.

이건 가정이 아닙니다. Feature 6 검증 중에 실제로 일어났습니다 — 같은 에이전트가 Feature 1부터 5까지는 전체 검증 프로토콜을 올바르게 실행한 후에요. 컨텍스트 압축이 verify-phases.md 내용을 먹어치웠고, 참조는 선택사항으로 취급되었습니다.

**에이전트의 회피 패턴은 놀라울 정도로 창의적입니다.** 관찰된 패턴 중 일부입니다:

- **"Health Check Passed" 회피**: 에이전트가 빌드 성공을 확인한 후 "health check가 통과했으므로 HARD STOP이 불필요하다"고 판단. 실제로는 빌드 성공과 기능 검증은 완전히 다른 것입니다. "코드가 컴파일된다"와 "기능이 올바르게 동작한다"는 별개인데, 에이전트는 "통과"라는 시그널 하나에 전체 검증을 생략합니다. 10개 SC 중 0개가 실제로 테스트된 상태에서 "verify 완료"가 보고됩니다.

- **"MANDATORY" 키워드의 무력화**: "MANDATORY"라고 쓰면 지켜질 거라 기대했지만, 컨텍스트가 압축되면 "MANDATORY RULE 3: Execute+Review"가 "review after execution"으로 요약됩니다. "review after execution"은 지시가 아니라 메모처럼 읽히고, 에이전트는 건너뜁니다.

- **"Do NOT skip"도 skip**: "Do NOT skip this step"이라고 명시적으로 쓴 지시가, 긴 대화 끝에 "verification notes: some steps may be skipped if passing"으로 압축됩니다. 원래 지시의 정반대가 된 겁니다.

이 패턴들을 관찰하면서 규칙의 강도를 점진적으로 높여야 했습니다. 진화 과정은 이랬습니다:

1. **가이드라인** ("~해야 합니다") — 처음 몇 번은 따르다가, 컨텍스트가 길어지면 무시됨
2. **경고** ("⚠️ 필수입니다") — 에이전트가 경고를 인식하지만 "이번에는 해당 없다"고 합리화
3. **강한 경고** ("🚨 절대 건너뛰지 마세요") — 2~3번 작동 후, 컨텍스트 압축되면 경고 자체가 사라짐
4. **BLOCKING gate** ("🚫 이것 없이 다음 단계 진행 불가") — 지속적으로 작동. 다음 단계의 입력이 구조적으로 이 단계의 출력에 의존하므로, 건너뛸 수 없음

단계 1에서 4까지 도달하는 데 수십 번의 파이프라인 실행이 필요했습니다. 매번 "이번엔 충분히 강하게 썼다"고 생각했지만, 에이전트는 늘 새로운 합리화 방법을 찾았습니다.

해결에는 세 가지 레이어가 필요했습니다:

**레이어 1 — 인라인 지시.** 규칙이 참조된 파일이 아니라 실행 지점에 직접 나타납니다. "verify-phases.md 참조"가 아니라 실제 규칙이 바로 거기에.

**레이어 2 — 차단 게이트.** 경고("⚠️ 검증해야 합니다")가 아니라 차단("🚫 BLOCKING: 검증 미완료. 머지로 진행 불가."). 차이는 에이전트가 차단을 합리화해서 넘어갈 수 없다는 것 — 검증을 완료하지 않으면 구조적으로 다음 단계의 출력을 생산할 수 없습니다.

**레이어 3 — 안티패턴 예시.** 명시적 WRONG과 RIGHT 패턴:

```
❌ WRONG: 빌드+TS 실행, "verify ✅" 표시, 머지로 진행
   → 10개 SC 미검증. Feature가 UI에서 도달 불가.

✅ RIGHT: verify-phases.md 읽기 → Phase 0-4 실행 →
   SC 커버리지 매트릭스 표시 → AskUserQuestion
```

이것이 spec-kit-skills에 중복처럼 보이는 텍스트가 있는 이유입니다. HARD STOP 재질문 지시가 여러 파일에 걸쳐 30회 이상 나타납니다. 각각이 다른 실행 지점에 있습니다. 공유 파일로 추출하면 "더 깔끔한" 코드가 되겠지만 — 에이전트에게 무시됩니다.

**Sub-Skill Call 함정:**

특히 교활했던 문제 하나가 있습니다. spec-kit-skills 초기에 파이프라인의 각 단계를 `Skill(speckit-specify)`처럼 sub-skill로 호출했습니다. 깔끔한 설계였죠 — 각 단계가 독립적이고, 재사용 가능하고, 관심사가 분리되어 있으니까요.

문제는: **Skill tool 호출이 응답 경계를 만든다**는 것이었습니다. `Skill(speckit-specify)`가 완료되면, 그 스킬의 completion 메시지가 최종 응답이 됩니다. 상위 파이프라인으로 제어가 돌아오지 않습니다. 즉, Execute 후에 와야 하는 Review — 아티팩트 읽기, 요약 표시, AskUserQuestion 호출 — 가 실행되지 않습니다. 사용자는 spec-kit의 raw output만 보고, 뭘 해야 하는지 모르는 채로 멈추게 됩니다.

해결책은 **Inline Execution**이었습니다. `Skill(speckit-specify)`를 호출하는 대신, 해당 SKILL.md를 직접 읽고 그 단계를 인라인으로 실행합니다. 이렇게 하면 제어가 상위 파이프라인에 남아있으므로, Execute 직후에 Review를 이어서 수행할 수 있습니다.

이것은 소프트웨어 엔지니어링의 직관과 정면으로 충돌합니다. "함수 호출 대신 코드를 인라인하라"는 조언은 일반적으로 안티패턴입니다. 하지만 에이전트 스킬에서는 제어 흐름의 연속성이 코드 재사용보다 중요합니다. 응답 경계에서 끊기면 사용자가 파이프라인의 어디에 있는지 모르게 되니까요.

이 문제의 근본 원인을 더 파헤쳐 봅시다. 일반 프로그래밍에서 함수를 호출하면 호출자에게 제어가 반환됩니다 — 이건 언어가 보장합니다. 하지만 Skill tool은 다릅니다. `Skill(speckit-specify)`를 호출하면 그 스킬이 자체적으로 응답을 생성하고, 그 응답이 사용자에게 직접 전달됩니다. 호출자(smart-sdd 파이프라인)에게 제어가 반환되지 않습니다. 마치 함수 호출이 `return`이 아니라 `process.exit()`으로 끝나는 것과 같습니다. 따라서 "호출 후에 해야 할 일"(Review, AskUserQuestion)을 호출자 측에 둘 수 없습니다 — 호출자가 실행되지 않으니까요.

Inline Execution은 이 문제를 우회합니다. "함수를 호출하는" 대신 "함수의 코드를 복사해서 현재 컨텍스트에서 실행하는" 방식입니다. 우아하지 않지만, 제어 흐름이 끊기지 않는다는 보장을 얻습니다.

**패턴의 또 다른 구체적 예시:**

에이전트가 종종 spec-kit 커맨드를 실행하고, raw output을 보여주고, 거기서 멈추는 것을 발견했습니다 — 생성된 아티팩트를 읽지도, 리뷰를 표시하지도, 사용자 승인을 요청하지도 않고 (일명 "Skill Tool Response Boundary" 문제). 해결은 단일 규칙이 아니라 4중 방어였습니다:

1. SKILL.md (항상 로드)에 MANDATORY RULE 3: "모든 speckit-* 실행 후 반드시 아티팩트를 읽고, 리뷰를 보여주고, AskUserQuestion을 호출해야 합니다"
2. 각 파이프라인 단계의 섹션에 해당 단계에 특화된 인라인 Execute+Review 프로토콜
3. Catch-all fallback: 어떤 이유로든 응답이 AskUserQuestion 없이 끝나면 continue 프롬프트를 표시
4. raw spec-kit 탐색 메시지가 누출되는 것을 감지하는 Stop 훅 (셸 스크립트)

왜 4중 방어가 필요한지 의아할 수 있습니다. 레이어 1만으로 충분하지 않을까? 답은 P2 원칙에 있습니다: 단일 레이어는 컨텍스트 압축에 취약합니다. MANDATORY RULE 3이 압축되면 레이어 1이 사라집니다. 하지만 레이어 2(각 단계별 인라인 프로토콜)는 해당 단계의 지시에 포함되어 있으므로, 그 단계를 실행하는 시점에는 항상 컨텍스트에 존재합니다. 레이어 3(catch-all fallback)은 "모든 다른 방어가 실패해도"를 커버합니다. 레이어 4(Stop 훅)는 에이전트 밖의 시스템 레벨 방어입니다. 각 레이어가 다른 실패 모드를 커버하므로, 모든 레이어가 동시에 실패하지 않는 한 시스템이 올바르게 동작합니다.

### P3: File over Memory — 영속성의 원칙

> 모든 중간 산출물과 상태는 파일에 저장한다. 에이전트 메모리에 의존하지 않는다.

에이전트의 컨텍스트 윈도우를 상태 저장에 신뢰할 수 없게 만드는 세 가지 속성:

- **유한** — 대화가 길어지면 압축됩니다. 50개 메시지 후, 초기 컨텍스트가 요약되거나 완전히 삭제됩니다.
- **세션 범위** — 세션이 끝나면 모든 것이 사라집니다. 다음 세션은 백지에서 시작합니다.
- **검사 불가** — 에이전트가 무엇을 기억하는지 볼 수 없습니다. "왜 Feature 1이 UUID를 사용한다는 걸 잊었을까?"를 디버깅할 수 없습니다.

파일에는 이런 문제가 없습니다. 영구적이고, diff 가능하고 (`git diff`로 정확히 무엇이 변했는지 볼 수 있음), 편집 가능하고 (스펙을 수동으로 고치고 재실행할 수 있음), 공유 가능합니다 (다른 에이전트나 사람이 작업을 이어갈 수 있음).

`sdd-state.md`가 상태 머신 파일로 존재하는 이유가 이것입니다. 컨텍스트 압축이나 새 세션 후 에이전트가 재개할 때, 파일을 읽고, "F003은 implement 단계, 태스크 4/7"을 보고, 거기서 계속합니다. 추측 없이, "우리가 뭘 하고 있었더라..."도 없이.

**verify 진행 상태 복구가 이 원칙의 진가를 보여주는 사례입니다.** Feature 6 검증 중이었습니다. Phase 0(pre-flight)과 Phase 1(Build + TypeScript)을 통과하고, Phase 2(SC 기능 검증)에서 10개 SC 중 7개까지 검증을 완료한 시점에 컨텍스트 압축이 발생했습니다. 상태 파일이 없었다면? 에이전트가 "verify를 해야 한다"는 것만 기억하고, build + TypeScript부터 다시 시작했을 겁니다 — 이미 검증한 7개 SC를 전부 다시 하면서요.

하지만 sdd-state.md에 verify 진행 상태가 phase 경계와 함께 기록되어 있었습니다. 에이전트가 상태 파일을 읽고, "Phase 2, SC 8/10부터 재개"를 확인하고, 정확히 거기서 이어갔습니다. 파일이 기억을 대체한 순간이었습니다.

이 원칙은 디버깅에서도 빛을 발합니다. 에이전트가 잘못된 결정을 내렸을 때, 파일에 기록된 상태를 보면 *어디서부터* 잘못되었는지 추적할 수 있습니다. sdd-state.md의 git log를 보면 "이 Feature가 specify에서 plan으로 넘어간 시점"이 정확히 기록되어 있고, 해당 시점의 spec.md diff를 보면 어떤 SC가 누락되었는지 알 수 있습니다. 에이전트 메모리에만 의존했다면 "어딘가에서 잘못되었는데 어디인지 모르겠다"가 됩니다.

P3의 또 다른 이점은 **사용자 개입의 용이성**입니다. 에이전트가 생성한 spec.md가 마음에 들지 않으면? 파일을 직접 열어서 수정하고, 파이프라인을 이어서 실행하면 됩니다. 에이전트가 수정된 스펙을 읽고 그 위에 plan을 세웁니다. 에이전트 메모리 기반이었다면 "내가 스펙을 이렇게 바꿨는데, 그걸 반영해서 다시 plan을 세워줘"라는 복잡한 대화가 필요했을 겁니다 — 그리고 에이전트가 원래 스펙의 잔재를 메모리에서 완전히 지우는지 보장할 수도 없습니다.

## 파일 아키텍처: 400개 이상의 파일이 컨텍스트 효율적으로 유지되는 방법

### 핵심 패턴

왜 400개 이상의 파일이 필요한지 의아할 수 있습니다. 답은: 조합 폭발(combinatorial explosion) 때문입니다. 9개 인터페이스, 47개 concern, 15개 archetype가 있으면, 가능한 프로젝트 프로필의 수는 천문학적입니다. 하나의 거대한 파일에 모든 규칙을 넣으면 두 가지 문제가 생깁니다: (1) 관리 불가능 — 하나의 규칙을 수정할 때 만 줄짜리 파일을 탐색해야 합니다. (2) 컨텍스트 낭비 — 에이전트가 불필요한 규칙까지 읽어야 합니다. 작은 모듈 파일로 분리하면, 각 파일이 하나의 관심사만 다루고, 필요할 때만 로드됩니다.

모든 스킬이 같은 구조를 따릅니다:

```
SKILL.md              — 항상 로드 (~200줄)
commands/             — 온디맨드 로드 (호출된 커맨드만)
reference/
  injection/          — 커맨드별 컨텍스트 조립 규칙
  state-schema.md     — 상태 머신 정의
  pipeline-integrity-guards.md
domains/
  _core.md            — 보편 규칙 (도메인과 함께 항상 로드)
  _resolver.md        — 모듈 로딩 로직
  interfaces/         — 9개 모듈 (gui, cli, http-api, grpc, tui, embedded, mobile, library, data-io)
  concerns/           — 47개 모듈 (auth, realtime, resilience, connection-pool, tls-management, ...)
  archetypes/         — 15개 모듈 (ai-assistant, microservice, network-server, ...)
  contexts/modes/     — 4개 모듈 (greenfield, rebuild, adoption, incremental)
  profiles/           — 15개 사전 빌드 프로필
```

### 이 구조가 동작하는 이유: 컨텍스트 예산

LLM 컨텍스트 윈도우는 크지만 무한하지 않습니다. 모든 것을 로드하면 에이전트가 작업을 시작하기도 전에 컨텍스트 예산을 소진합니다. spec-kit-skills는 **선택적 로딩** — 현재 태스크와 프로젝트 프로필에 맞는 모듈만 로드합니다.

일반적인 파이프라인 단계의 계산:

- **항상 로드:** ~200줄 (SKILL.md — 라우팅 + 필수 규칙)
- **커맨드별:** ~500줄 (특정 커맨드 파일, 예: pipeline.md)
- **도메인별:** ~400줄 (Domain Profile에 맞는 3~5개 모듈 x ~100줄)
- **Feature별:** ~200줄 (프리컨텍스트 + 현재 스펙)
- **합계:** ~1,300줄의 작업 컨텍스트

전부 로드하면: ~15,000줄 이상 — 에이전트가 작업 대신 지시를 읽는 데 컨텍스트 예산을 쓰게 됩니다. 더 심각한 문제는 양이 아니라 **노이즈**입니다. 15,000줄 중 현재 작업과 관련 없는 13,000줄이 에이전트의 주의를 분산시킵니다. cli 프로젝트인데 "모든 버튼에 호버 상태가 필요합니다"라는 gui 규칙이 컨텍스트에 있으면, 에이전트가 혼란스러워합니다 — "이 프로젝트에 버튼이 있나? 아마 나중에 필요하겠지..." 하면서 불필요한 SC를 생성하기도 합니다.

선택적 로딩이 이것을 실용적으로 만듭니다. 로드할 모듈을 결정하는 것은 에이전트가 아니라 **resolver**입니다. Domain Profile에 선언된 축을 읽고, 해당하는 모듈만 자동으로 로드합니다. 에이전트가 "어떤 모듈을 읽을까?"를 판단하지 않으므로, 잘못된 모듈을 선택하거나 필요한 모듈을 빠뜨릴 위험이 없습니다.

### 도메인 모듈 로딩 순서

파이프라인이 실행될 때, 모듈은 특정 순서로 로드됩니다. 각 레이어가 이전 것을 확장할 수 있습니다:

1. **_core.md** — 모든 프로젝트에 적용되는 보편 규칙
2. **interfaces/{name}.md** — Interface 축에 특화된 규칙 (gui, cli, http-api, grpc 등)
3. **concerns/{name}.md** — 각 활성 Concern에 대한 규칙 (auth, realtime, resilience 등)
4. **archetypes/{name}.md** — 도메인 철학 규칙 (ai-assistant, microservice, network-server 등)
5. **foundations/{framework}.md** — 프레임워크 특화 규칙 (electron, fastapi, go 등)
6. **org-convention.md** — 조직 전체 규칙 (선택사항, 프로젝트 간 공유)
7. **contexts/modes/{name}.md** — 생명주기 규칙 (greenfield, rebuild, adoption)
8. **domain-custom.md** — 프로젝트 수준 오버라이드 (선택사항)

모듈은 표준화된 섹션 번호를 사용합니다. Interface와 Concern은 S0~S9. Archetype는 A0~A5 (충돌을 피하기 위한 별도 번호 체계). 여러 모듈이 활성화되면, 해당 섹션이 **병합**됩니다 — `gui.md`의 S1 (SC 생성 규칙)이 `realtime.md`의 S1, `ai-assistant`의 A2 (SC 확장)와 함께 누적됩니다.

병합 의미론은 append 기반입니다: `gui`가 "모든 인터랙티브 요소에 호버 상태 SC가 필요하다"고 하고 `realtime`이 "모든 스트리밍 디스플레이에 완료 표시기 SC가 필요하다"고 하면, 결합된 S1 섹션에 양쪽 규칙이 모두 포함됩니다. 규칙이 다른 도메인에서 동작하므로 충돌이 없습니다.

"왜 오버라이드가 아니라 append인가?"라고 물을 수 있습니다. 초기에 오버라이드 방식을 시도했습니다 — 후순위 모듈이 선순위를 대체하는 방식. 문제는 의도치 않은 규칙 손실이었습니다. `realtime` 모듈이 `gui` 모듈의 S1을 오버라이드하면, gui의 폼 검증 규칙이 사라집니다. 둘 다 필요한데 하나만 남는 겁니다. Append 방식은 "모든 관련 규칙이 누적된다"는 보장을 줍니다. 규칙 간 실제 충돌이 있으면 (드물지만) project-level 오버라이드(domain-custom.md)에서 해결합니다.

### Context Injection: 핵심 혁신

시스템의 아키텍처적 심장입니다. 각 파이프라인 단계마다 컨텍스트 조립을 오케스트레이션하는 **injection 파일**이 있습니다:

```
reference/injection/
  specify.md    — speckit-specify가 받는 컨텍스트
  plan.md       — speckit-plan이 받는 컨텍스트
  implement.md  — speckit-implement가 받는 컨텍스트
```

자연어 규칙을 위한 의존성 주입 컨테이너라고 생각하세요. Spring의 `@Autowired`나 Angular의 서비스 프로바이더처럼, injection 파일은 규칙을 직접 담고 있지 않습니다 — 어떤 모듈을 로드할지, 어떤 GEL 아티팩트를 포함할지, 어떤 단계별 지시를 추가할지 선언합니다. 차이점은 주입되는 것이 코드가 아니라 자연어 규칙이라는 것입니다.

각 파이프라인 단계마다 다른 injection 파일이 활성화됩니다. specify 단계에서는 SC 생성 규칙(S1)이 핵심이고, plan 단계에서는 아키텍처 제약(S2)과 태스크 분해 규칙이 핵심이고, implement 단계에서는 프레임워크 관용구(F3)와 버그 방지 규칙(S7)이 핵심입니다. 같은 Domain Profile이라도 파이프라인 단계에 따라 다른 모듈 섹션이 로드됩니다.

**7단계 읽기 프로토콜을 따라가 봅시다.** ai-assistant + gui + realtime 프로젝트에서 `speckit-specify`가 실행될 때 실제로 무슨 일이 벌어지는지:

1. **Core 규칙 로드**: injection 파일이 `_core.md` § S1을 읽음 — 보편 SC 생성 규칙 (모든 프로젝트에 공통)
2. **Interface 규칙 로드**: `gui.md` § S1 — "모든 폼에 검증 피드백 SC, 모든 내비게이션에 라우트 가드 SC"
3. **Concern 규칙 로드**: `realtime.md` § S1 — "모든 스트리밍 디스플레이에: 시작 표시, 진행 업데이트, 완료 신호, 에러 폴백, 재연결 로직"
4. **Archetype 규칙 로드**: `ai-assistant` § A2 — "프로바이더 추상화에: 멀티 프로바이더 SC, Rate Limit 처리 SC, 토큰 예산 관리 SC"
5. **단계별 규칙 추가**: Brief → SC 매핑, 정교화 프로브, SC 번호 규칙
6. **GEL 컨텍스트 포함**: 기존 entity-registry.md, api-registry.md
7. **Feature 컨텍스트 포함**: 이 Feature의 pre-context.md

**Before/After를 비교해 봅시다.**

Context Injection 없이 (Before):
- 15,000줄의 규칙이 한꺼번에 로드됩니다
- 에이전트가 현재 Feature와 관련 있는 부분을 스스로 찾아야 합니다
- cli 프로젝트인데 gui 규칙을 적용하는 일이 빈번했습니다
- greenfield인데 rebuild 시나리오의 "기존 코드 보존" 규칙을 참조
- 에이전트의 컨텍스트 예산 중 절반 이상이 "읽기"에 소비되어, 실제 작업에 쓸 여유가 부족

Context Injection 후 (After):
- 1,300줄의 정확히 맞춤된 컨텍스트만 제공됩니다
- 불필요한 규칙이 존재하지 않으므로, 잘못된 규칙을 적용할 확률이 구조적으로 제거됩니다
- 에이전트가 규칙을 "선택"할 필요 없이, 제공된 모든 규칙이 현재 프로젝트에 해당
- 컨텍스트 예산의 대부분이 실제 스펙 작성, 플랜 수립, 구현에 사용 가능

이 차이를 체감하는 순간이 있었습니다. 같은 프로젝트에서 Context Injection 적용 전과 후의 specify 결과를 비교했을 때, 적용 전에는 "모든 API에 rate limiting을 고려하라"는 SC가 생성되었습니다 — 해당 프로젝트에 rate-limiting concern이 없는데도요. 에이전트가 15,000줄 속의 rate-limiting 규칙을 "이것도 적용해야 하나?"라고 판단한 겁니다. 적용 후에는 rate-limiting 규칙 자체가 컨텍스트에 없으므로 이런 혼란이 원천적으로 불가능해졌습니다.

결과: speckit-specify가 이 특정 프로젝트의 필요를 반영하는 맞춤 컨텍스트를 받습니다. 다른 프로젝트 — 예를 들어 cli + resilience + microservice — 는 같은 injection 메커니즘을 통해 완전히 다른 규칙을 받게 됩니다.

한 가지 더: Context Injection은 **Graceful Degradation**도 지원합니다. optional artifact(org-convention, pre-context 등)가 아직 존재하지 않을 때, injection 파일이 해당 단계를 건너뛰고 나머지 컨텍스트만으로 진행합니다. 새 프로젝트에서 entity-registry가 아직 없다면? injection이 "registry 없음 — 새 엔티티를 자유롭게 정의 가능"이라는 컨텍스트를 대신 제공합니다. 에이전트가 "파일을 찾을 수 없다"는 에러에 멈추지 않고, 현재 상태에 맞게 적응합니다.

이것은 프로젝트 생명주기의 초기 단계에서 특히 중요합니다. 첫 번째 Feature를 시작할 때는 GEL 아티팩트가 거의 비어있습니다 — entity-registry에 엔티티가 없고, api-registry에 API가 없고, roadmap에 의존성이 없습니다. Graceful Degradation이 없으면 에이전트가 "필수 파일이 비어있다"는 이유로 멈출 수 있습니다. 대신, injection 시스템이 "이것은 첫 Feature이므로, 자유롭게 새 엔티티와 API를 정의하라"는 맞춤 컨텍스트를 제공합니다. 프로젝트가 성장하면서 GEL이 채워지면, 같은 injection 메커니즘이 "기존 엔티티를 재사용하라"는 컨텍스트로 자연스럽게 전환됩니다.

## 확장성: Convention over Configuration

### 새 Concern 모듈 추가

기존 concern에 맞지 않는 횡단 패턴을 발견했다고 합시다 — 예를 들어 `rate-limiting`을 독립적인 concern으로. 하는 일은:

1. `domains/concerns/rate-limiting.md` 생성
2. 스키마에 따라 섹션 추가:
   - S0: Signal Keywords (코드에서 이 concern을 감지하는 방법)
   - S1: SC Generation Rules (이 concern이 활성화되면 어떤 SC를 생성할지)
   - S3: Verify Steps (이 concern이 활성화되면 추가할 검증)
   - S5: Elaboration Probes (Brief 상담 중 물어볼 질문)
   - S7: Bug Prevention Rules (이 concern의 흔한 안티패턴)
3. 끝입니다. 사용자의 Domain Profile에 `rate-limiting`이 포함되면 resolver가 자동으로 발견합니다.

코드 변경 없음. 등록 단계 없음. 설정 파일 업데이트 없음. 예상 경로에 파일이 존재하면 로드됩니다. 이것이 "Convention over Configuration"의 실현입니다 — 파일의 위치와 이름이 곧 설정입니다. `domains/concerns/` 디렉토리에 있으면 concern 모듈이고, 파일명이 곧 concern 이름입니다. 별도의 매핑 테이블이나 등록 과정이 필요 없습니다. Ruby on Rails의 "파일이 특정 위치에 있으면 자동으로 로드된다"는 철학과 동일합니다.

이 설계 선택에는 중요한 이유가 있습니다: 등록 단계가 있으면 그것도 에이전트 지시로 작성해야 하고, 에이전트가 그 지시를 건너뛸 수 있습니다. "파일이 존재하면 자동으로 발견됨"은 에이전트가 건너뛸 수 없는 메커니즘입니다. 파일 시스템 자체가 레지스트리 역할을 합니다.

또한 이 구조는 **기여의 진입 장벽**을 낮춥니다. 새 concern 모듈을 추가하려는 사람은 기존 concern 파일 하나를 복사하고, 내용을 수정하고, 올바른 디렉토리에 넣으면 됩니다. 프레임워크의 내부 동작을 이해할 필요가 없습니다. 스키마(S0~S9 섹션 구조)만 따르면 됩니다.

### 새 Foundation 추가

프레임워크 특화 규칙은 `domains/foundations/{framework}.md`에 두 가지 섹션으로 존재합니다:

- **F2** — Specify-time 규칙 (이 프레임워크가 스펙 생성에 부과하는 제약)
- **F3** — Implement-time 규칙 (프레임워크 관용구, 안티패턴, 도구 체인 설정)

예를 들어, `electron.md`의 F2에는 "메인 프로세스가 렌더러 크래시에서 살아남아야 한다"가 필수 아키텍처 원칙으로 포함됩니다. F3에는 "Playwright에 `_electron.launch()` 사용, 사용자 설정 보존을 위해 `--user-data-dir` 전달" 같은 내용이 있습니다.

여기서 중요한 설계 원칙 하나: Foundation의 규칙은 **프레임워크 특화**이지만, 다른 모든 규칙(core, interface, concern, archetype)은 **프레임워크 독립적**이어야 합니다. "저장 데이터는 불변이고, 표시 시점에서만 변환한다"는 보편 규칙입니다. "React에서 `useMemo`로 변환한다"는 Foundation 규칙입니다. 이 분리가 없으면 React 프로젝트의 규칙이 다른 프레임워크에도 새어 들어갑니다.

이 원칙을 어긴 초기의 실패가 있었습니다. concern 모듈 중 하나에 "React의 `useEffect` cleanup에서 WebSocket을 닫으라"는 규칙이 있었습니다. 이 규칙은 React 프로젝트에서는 완벽하지만, Vue 프로젝트에서는 의미가 없고, Go 서버 프로젝트에서는 혼란만 줍니다. 수정 후에는 concern에 "컴포넌트 언마운트 시 연결을 정리하라"라는 보편 원칙만 두고, React/Vue/Svelte 각각의 구현은 Foundation에 둡니다.

### 커스텀 프로필 생성

프로필은 ~10줄의 매니페스트입니다:

```markdown
# Profile: my-ai-chat

interfaces: gui, http-api
concerns: auth, realtime, ai-assistants, external-sdk
archetype: ai-assistant
foundation: nextjs
scenario: greenfield
scale:
  project_maturity: mvp
  team_context: solo
```

`domains/profiles/my-ai-chat.md`에 저장하고, `/smart-sdd init --profile my-ai-chat`로 사용합니다. resolver가 프로필을 읽고 선언된 모든 모듈을 로드합니다. 15개의 사전 빌드 프로필이 제공되므로, 대부분의 프로젝트는 기존 프로필을 선택하거나 약간 수정하는 것만으로 시작할 수 있습니다.

### Scale: Context 축의 깊이 조절 컴포넌트

프로필에서 눈여겨볼 부분이 `scale` 섹션입니다. 이것은 Context 축(5번째 축)의 일부로, 다른 4개 축(Interface, Concern, Archetype, Foundation)이 "무엇을 적용할지"를 결정한다면, Scale은 "얼마나 깊이 적용할지"를 조절합니다.

**Project Maturity** — 프로젝트의 성숙도에 따라 SC의 깊이와 검증의 엄격함을 조절합니다:

- `prototype`: 최소 SC, 검증 간소화. "빨리 돌아가는 것을 보자"에 최적화. 에러 핸들링 SC는 생략 가능하고, happy path 중심으로 진행합니다. 스펙도 간결하게 — 핵심 기능만 정의합니다.
- `mvp`: 핵심 SC + 에러 핸들링. "사용자에게 보여줄 수 있는 수준." 주요 에러 시나리오는 커버하되, 극단적 엣지 케이스는 후순위. 로딩 상태와 에러 메시지는 포함하지만, 오프라인 지원이나 접근성은 선택적입니다.
- `production`: 전체 SC + 엣지 케이스 + 성능 고려. "실 서비스에 배포 가능한 수준." 모든 에러 핸들링, 접근성, 보안 검증이 활성화. 성능 관련 SC(로딩 최적화, 캐싱 전략)도 포함됩니다.

**Team Context** — `solo`, `small-team`, `large-team` 중 하나:

- `solo`: 코드 리뷰 단계 간소화. 본인이 모든 맥락을 알고 있으므로, 내부 구현 문서보다 동작하는 코드에 집중. verify에서도 형식적 SC 커버리지보다 실제 동작 검증에 무게를 둡니다.
- `small-team`: 기본 컨벤션 검증 활성화. API 계약의 명시적 문서화, PR 설명 수준의 의사결정 기록 요구. 팀원이 코드 리뷰 시 "왜 이렇게 했는지"를 spec.md에서 찾을 수 있어야 합니다. naming convention 검증도 활성화되어, 팀 내 일관성이 유지됩니다.
- `large-team`: 상세한 API 계약 문서화, 엄격한 인터페이스 경계 검증, 마이그레이션 가이드 생성. 다른 팀이나 다른 조직의 개발자가 맥락 없이 읽어도 통합할 수 있는 수준. entity-registry와 api-registry의 변경 사항에 대한 호환성 검증도 추가됩니다.

같은 `gui + realtime + ai-assistant` 프로필이라도, `prototype + solo`면 15개 SC가 생성되고, `production + large-team`이면 40개 이상의 SC가 에러 핸들링, 접근성, 계약 문서와 함께 생성됩니다. 이것이 중요한 이유는, 프로토타입 단계에서 production 수준의 SC를 강제하면 에이전트가 과도한 엣지 케이스 처리에 시간을 소비하게 되고, 반대로 production에서 prototype 수준의 SC만 생성하면 실 사용자에게 영향을 미치는 에러 시나리오가 누락됩니다. Context scale은 이 균형을 프로필 차원에서 관리합니다.

Context scale은 프로젝트 진행에 따라 변경할 수 있습니다. prototype 단계에서 시작해서, MVP로 전환할 때 `project_maturity: mvp`로 업데이트하면, 기존 Feature의 스펙에 에러 핸들링 SC가 추가로 필요한지 자동으로 감지됩니다. `production`으로 전환하면 접근성(accessibility) 관련 SC와 성능 관련 SC가 요구 목록에 추가됩니다. 프로젝트 초기의 빠른 이터레이션과, 출시 전의 철저한 품질 보증을 같은 프레임워크 안에서 관리할 수 있습니다.

### 3단계 규칙 계층

규칙은 세 가지 수준에서 올 수 있으며, 후순위가 선순위를 오버라이드합니다:

1. **스킬 수준** — `_core.md` + 도메인 모듈 (보편적, 프로젝트가 관리)
2. **조직 수준** — `org-convention.md` (조직의 프로젝트 전체에 공유)
3. **프로젝트 수준** — `domain-custom.md` (하나의 프로젝트에 특화)

이는 회사가 조직 전체 규칙(네이밍 패턴, API 설계 표준, 보안 요구사항)을 정의하면 spec-kit-skills를 사용하는 모든 프로젝트에 자동 적용되고, 개별 프로젝트는 특정 규칙을 오버라이드할 수 있다는 뜻입니다.

예를 들어, 조직 규칙에 "모든 API 응답에 request-id 헤더를 포함한다"가 있고, 특정 프로젝트에서 "내부 마이크로서비스 간 통신에는 request-id를 생략한다"로 오버라이드한다면, 해당 프로젝트에서는 오버라이드가 적용되지만 다른 프로젝트에서는 원래 조직 규칙이 유지됩니다. 이 계층 구조는 CSS의 specificity와 비슷합니다 — 더 구체적인 규칙이 더 일반적인 규칙을 오버라이드합니다.

## Pipeline Integrity Guards

7개의 가드가 파이프라인의 정확성을 강제합니다. 각각 특정 전환 지점에서의 차단 검사입니다 — 일반 소프트웨어의 pre-commit hook이나 CI/CD gate와 같은 역할이지만, 검사 대상이 코드가 아니라 "에이전트가 올바른 컨텍스트를 가지고 있는가"입니다. 각 가드가 없을 때 실제로 어떤 일이 벌어지는지 보겠습니다:

**Constitution Guard.** specify 전: 헌법이 정의되어 있는가? 없으면 스펙에 지도 원칙이 없습니다. 헌법(constitution)은 프로젝트의 핵심 아키텍처 원칙과 기술적 의사결정을 담고 있습니다 — "왜 이 기술 스택을 선택했는지", "데이터 저장 전략은 무엇인지", "보안 원칙은 무엇인지". 실제 실패: 헌법 없이 specify를 실행하자, 에이전트가 자체 판단으로 아키텍처 결정을 내렸습니다. "왜 서버 상태를 파일 시스템 대신 SQLite에 저장하나요?"라는 질문에 답이 없었습니다 — 지도 원칙이 없었으니까요. 헌법에 "임베디드 환경에서 외부 의존성을 최소화한다"는 원칙이 있었다면, SQLite 대신 파일 시스템이라는 일관된 결정이 나왔을 겁니다.

**Entity Registry Guard.** plan 전: 스펙의 모든 엔티티가 등록되었는가? 없으면 플랜이 기존 데이터 모델을 참조할 수 없습니다. 실제 실패: Feature 2의 plan이 Feature 1에서 정의한 Document 엔티티와 다른 스키마의 Document를 참조. Feature 1의 Document에는 `createdAt: Date`가 있었고 Feature 2는 `created: string`으로 정의. 통합 시 필드 충돌과 타입 불일치가 발생했고, ORM 레벨에서 마이그레이션 충돌까지 이어졌습니다.

**API Registry Guard.** implement 전: 플랜의 모든 API가 등록되었는가? 없으면 구현이 계약을 추측합니다. 실제 실패: 프론트엔드가 `GET /api/documents?status=active`를 호출했는데, 백엔드는 `GET /api/documents?filter=active`로 구현. 등록된 계약이 없어 양쪽이 각자의 추측대로 구현. 이 종류의 버그는 타입 체크나 빌드로는 잡히지 않습니다 — 런타임에 "documents가 로드되지 않는다"로 나타나고, 원인을 추적하는 데 시간이 소요됩니다.

**Pre-Context Guard.** specify 전: 이 Feature의 프리컨텍스트가 존재하는가? 없으면 스펙의 근거가 없습니다. 실제 실패: pre-context 없이 "채팅 기능 추가"를 specify하자, 에이전트가 일반적인 채팅 앱을 상상해서 스펙을 썼습니다 — 1:1 메시지, 그룹 채팅, 읽음 표시까지 포함해서요. 실제 앱은 AI 어시스턴트여서 필요한 건 "사용자-AI 대화 인터페이스"였는데, 완전히 다른 방향의 메신저 스펙이 나온 겁니다. pre-context가 "이 앱은 AI 어시스턴트이며, 채팅은 사용자와 AI 간의 대화를 의미한다"고 명시했다면 이런 일은 발생하지 않았을 겁니다.

**Dependency Order Guard.** pipeline 전: 이 Feature의 의존성이 완료되었는가? 없으면 구현이 아직 존재하지 않는 엔티티를 참조합니다. 실제 실패: Feature 3(문서 공유)이 Feature 1(사용자 관리)보다 먼저 구현됨. User 엔티티가 없어서 공유 로직에 하드코딩된 사용자 ID가 들어감. Feature 1이 나중에 구현되었을 때, Feature 3의 하드코딩된 부분을 전부 찾아 교체해야 했고, 일부는 놓쳐서 런타임 에러로 이어졌습니다.

**Augmentation Guard.** `add --to` 후: 파이프라인이 재실행되었는가? 없으면 스펙이 새 요구사항을 반영하지 않습니다. 실제 실패: 기존 Feature에 "오프라인 지원" 요구사항을 추가했지만, 파이프라인을 재실행하지 않아 기존 스펙/플랜이 온라인 전용 가정을 유지. 구현 단계에서 오프라인 관련 코드가 전혀 없는 채로 "구현 완료"가 보고되었습니다.

**Regression Guard.** verify에서 이슈 발견 후: 회귀가 해결되었는가? 없으면 알려진 버그가 이월됩니다. 실제 실패: verify에서 "사이드바가 화면 넘침"을 발견하고 기록했지만, 다음 Feature에서 그 수정 없이 빌드를 이어감. 3개 Feature 후에도 같은 버그가 남아있음. 더 나쁜 건, 후속 Feature의 레이아웃이 그 깨진 사이드바를 "정상"으로 간주하고 그 위에 구축되어, 버그 수정 시 연쇄적 레이아웃 붕괴가 발생했다는 겁니다.

각 가드는 같은 패턴을 따릅니다: 조건 확인 → 실패하면 차단 메시지 표시 → 조건이 충족될 때까지 에이전트 진행 불가. 이 가드들은 경고가 아닙니다 — P2 원칙(Enforce, Don't Reference)에 따라, 조건이 충족되지 않으면 파이프라인이 **구조적으로 진행할 수 없도록** 설계되어 있습니다. 에이전트가 "이번에는 괜찮겠지"라고 합리화할 여지를 주지 않습니다.

### Bug Fix Severity 시스템

verify 단계에서 버그를 발견했을 때, 모든 버그를 같은 방식으로 처리하면 안 됩니다. 사소한 CSS 정렬 문제와 데이터 모델 설계 결함은 근본적으로 다른 수준의 대응이 필요합니다. 이를 위해 4단계 심각도 시스템을 사용합니다:

- **Minor**: verify 단계에서 즉시 수정 가능. 오타, 색상 값, 간격 조정 등. verify를 벗어나지 않습니다.
- **Major-Implement**: implement 단계로 되돌아가야 하는 수준. 로직 버그, 누락된 에러 핸들링 등. 스펙과 플랜은 올바르지만 구현이 잘못된 경우.
- **Major-Plan**: plan 단계로 되돌아가야 하는 수준. 태스크 분해가 잘못되었거나, 의존성 순서가 틀렸거나, 아키텍처 접근이 잘못된 경우.
- **Major-Spec**: specify 단계로 되돌아가야 하는 수준. SC 자체가 잘못되었거나, 요구사항이 누락되었거나, 근본적인 설계 결함인 경우. 예를 들어 "사용자가 문서를 삭제할 수 있다"는 SC가 있는데, 실제로는 "소프트 딜리트여야 한다"는 요구사항이 누락된 경우 — 이건 구현을 고치는 수준이 아니라, 스펙 자체를 수정해야 합니다.

Minor만 verify에서 바로 고칩니다. 나머지는 해당 단계로 정확히 되돌아갑니다. "일단 여기서 고쳐보자"는 유혹을 구조적으로 차단합니다.

왜 이게 필요할까요? verify 단계에서 에이전트가 데이터 모델 설계 결함을 발견했을 때, verify 안에서 "빠르게" 고치려고 하면 스펙과 구현 사이에 불일치가 생깁니다. 코드는 수정되었지만 spec.md는 여전히 원래 설계를 기술하고 있으니까요. 다음 Feature가 이 스펙을 참조하면 잘못된 설계를 기반으로 작업하게 됩니다. 심각도에 따라 올바른 단계로 되돌아가면, 스펙 → 플랜 → 구현 → 검증의 일관성이 유지됩니다.

## 아키텍처를 돌아보며

여기까지 읽으셨다면 한 가지 의문이 들 수 있습니다: "이 모든 복잡성이 정말 필요한가?" 400개 이상의 마크다운 파일, 7개의 파이프라인 가드, 46개의 조합 패턴, 30회 이상 반복되는 HARD STOP 텍스트.

솔직한 답은: 처음부터 이렇게 설계한 게 아닙니다.

처음에는 SKILL.md 하나에 모든 규칙을 넣었습니다. 에이전트가 무시했습니다.

파일을 분리했습니다. 에이전트가 참조를 건너뛰었습니다.

BLOCKING gate를 추가했습니다. 에이전트가 새로운 회피 패턴을 찾았습니다.

매번 실패할 때마다 한 겹의 방어가 추가되었고, 그 결과가 현재의 시스템입니다.

이것은 "과설계"가 아니라 "실패 주도 설계(failure-driven design)"입니다. 모든 규칙, 모든 가드, 모든 반복된 텍스트 뒤에는 실제로 발생한 실패가 있습니다. 그리고 이 실패들은 spec-kit-skills에만 해당하는 게 아닙니다 — AI 에이전트를 위한 도구를 만드는 모든 사람이 결국 같은 문제를 만나게 됩니다. 컨텍스트 압축, 참조 무시, 상태 손실은 현재 LLM 기반 에이전트의 구조적 한계이며, 이 한계 위에서 안정적인 시스템을 구축하는 것이 이 시대의 새로운 엔지니어링 도전입니다.

4편에서는 이 실패들을 체계적으로 정리합니다: 19개의 갭 패턴, 50개 이상의 구체적 교훈. "Context Compression이 규칙을 삼켰다", "에이전트가 참조를 무시했다", "Feature 간 정보가 단절되었다" — 여기서 다룬 아키텍처의 모든 요소가 구체적인 실패 사례에서 어떻게 태어났는지의 전체 이야기입니다.

## 🤖 에이전트를 위한 글 — 아키텍처 참조

> 위의 모든 내용은 *왜, 어떻게*의 이야기였습니다. 아래의 모든 내용은 *같은 정보*를 AI 에이전트가 직접 소비할 수 있는 형식으로 인코딩합니다 — 파일 경로, 로딩 순서, 섹션 스키마, 가드 조건. 자체 스킬 시스템을 구축한다면 이것이 사양서입니다. 복사하고, 적응시키고, 체크리스트로 사용하세요.

```
architecture:
  principles:
    P1 Context Continuity:
      P1-a: Domain Profile이 first-class (모듈 섹션을 통해 모든 파이프라인 단계를 형성)
        cross_concern_integration: 46개 조합 패턴 (gui+realtime → optimistic update, reconnection UI, stale prevention)
      P1-b: Artifact Separation (스펙은 무엇을 기술, 어디서 왔는지는 기술하지 않음)
        source_code_reference: implement 시 pre-context의 행동적 본질을 BLOCKING gate로 재로드
        실패_사례: ModelSelector → SBI 기록 → implement에서 3개 텍스트 입력으로 퇴화
      P1-c: GEL 파일을 통한 Cross-Feature Memory (entity-registry, api-registry, sdd-state)
        GEL_없으면: Feature 간 엔티티 재발명, API 충돌, 타입 불일치

    P2 Enforce Don't Reference:
      요구사항: 모든 핵심 규칙에 3개 레이어 필요
        레이어 1: 실행 지점에 인라인 지시
        레이어 2: 차단 게이트 (경고가 아님 — 구조적 차단)
        레이어 3: 안티패턴 예시 (WRONG 다음 RIGHT)
      회피_패턴:
        - "health check 통과 → HARD STOP 불필요" 판단
        - "MANDATORY" 키워드가 압축 시 suggestion으로 변질
        - "Do NOT skip"이 압축 시 "may be skipped"으로 반전
      진화: guidelines → warnings → BLOCKING gates
      sub_skill_trap: Skill tool → 응답 경계 → Review 미실행 → 해결: Inline Execution
      증거: HARD STOP 텍스트가 30회 이상 등장 (각각 다른 실행 지점)
      이유: 에이전트가 참조를 선택적 읽기로 취급
      따름정리: 에이전트 스킬에서 DRY는 컴플라이언스를 죽임

    P3 File over Memory:
      요구사항: 모든 상태를 파일에 (sdd-state.md, 레지스트리, 프리컨텍스트)
      이유: 컨텍스트 윈도우는 유한하고, 세션 범위이며, 검사 불가
      이점: git diff = 완전한 감사 추적, 세션 간 영속, 사람이 편집 가능
      verify_복구: phase 경계와 SC 진행 상태를 파일에 기록 → 컨텍스트 압축 후 정확한 지점에서 재개

  파일 구조:
    SKILL.md: ~200줄, 항상 로드 (라우팅 + 필수 규칙)
    commands/: 호출별 로드 (~500줄)
    reference/injection/: 파이프라인 단계별 컨텍스트 조립 (자연어 규칙의 DI 컨테이너)
    domains/: Domain Profile별 로드 (3~5개 모듈에 ~400줄)

  도메인 모듈 시스템:
    로딩 순서: _core → interfaces → concerns → archetypes → foundations → org → context modes → context modifiers → custom
    섹션 스키마:
      interfaces/concerns: S0 (키워드), S1 (SC 규칙), S3 (검증), S5 (프로브), S7 (버그), S8 (런타임)
      archetypes: A0 (키워드), A1 (철학), A2 (SC 확장), A3 (프로브), A4 (헌법), A5 (Brief 기준)
    병합 규칙: append 의미론 (누적, 오버라이드 아님)
    선택: 활성 Domain Profile에 맞는 모듈만

  context_injection:
    7단계_프로토콜: core → interface → concern → archetype → step_rules → GEL → feature_context
    효과: 15,000줄 → 1,300줄 맞춤 컨텍스트. 잘못된 규칙 적용 구조적 제거.

  컨텍스트 예산:

  파이프라인 가드: 7개 (constitution, entity_registry, api_registry, pre_context,
                   dependency_order, augmentation, regression)
    패턴: 조건 확인 → 실패 시 → 차단 메시지 → 진행 불가
    각 가드별 실패 시나리오 문서화됨

  bug_fix_severity: 4단계 (Minor, Major-Implement, Major-Plan, Major-Spec)
    Minor만 verify에서 즉시 수정. 나머지는 해당 단계로 되돌림.

  확장성:
    concern 추가: domains/concerns/{name}.md에 S0~S9 작성 → 자동 발견
    foundation 추가: domains/foundations/{name}.md에 F2~F3 작성 → 자동 로드
    profile 추가: domains/profiles/{name}.md → --profile로 사용
    domain_extend: /domain-extend 스킬로 모듈 생성, ADR/문서 임포트, 검증 자동화
    scale_modifier:
      project_maturity: prototype (최소 SC) / mvp (핵심+에러) / production (전체+엣지케이스)
      team_context: solo (간소화) / small-team (기본 컨벤션) / large-team (상세 계약 문서화)
    규칙 계층: 스킬 수준 → 조직 수준 → 프로젝트 수준 (후순위가 선순위 오버라이드)
```

---

이 아키텍처 — 세 가지 철학, 일곱 개의 가드, 선택적 모듈 로딩, 컨텍스트 인젝션 — 는 계속 진화하는 시스템의 현재 상태입니다. 하지만 변하지 않은 것이 있습니다: 세 가지 원칙. Context Continuity, Enforce Don't Reference, File over Memory는 모든 반복을 통해 유지되었습니다. 이것을 구현하는 메커니즘은 계속 개선됩니다. 원칙은 수정이 필요하지 않았습니다.

결국 "왜 400개 파일인가?"의 답은 이것입니다 — 통제 가능한 AI 개발은 완벽한 프롬프트 하나를 쓰는 것이 아닙니다. 올바른 행동이 에이전트의 선의가 아니라 구조에서 나오는 시스템을 만드는 것입니다. 에이전트는 놀랍도록 유능합니다. 400개 파일은 그 유능함이 *당신의* 목표, *당신의* 아키텍처, *당신의* 품질 기준으로 향하게 합니다.

에이전트 코딩 도구가 더 똑똑해질수록 — 그리고 그럴 것입니다 — 이런 하네스의 가치는 더 커집니다. 가드레일 없이 더 유능한 에이전트는 리뷰하기 더 어려운, 더 인상적으로 보이는 코드를 만듭니다. 가드레일 *있는* 더 유능한 에이전트는 더 좋은 코드를, 더 빠르게, 완전한 추적성과 함께 만듭니다.
