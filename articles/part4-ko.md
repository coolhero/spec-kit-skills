## 에이전트 스킬 개발에 대한 솔직한 이야기

spec-kit-skills를 만드는 데 3주에 걸쳐 200개 이상의 커밋이 필요했습니다. 그 과정에서 AI 에이전트를 *신뢰할 수 있게* 만드는 것이 *유능하게* 만드는 것과 근본적으로 다르다는 것을 깨달았습니다.

에이전트는 항상 Feature를 만들 수 있었습니다. 문제는 *올바른* Feature를, *올바른 방식*으로, *매번* 만들도록 하는 것이었습니다.

## 실패 패턴

### 패턴 1: 빌드 통과 ≠ 기능 동작

**함정**: 빌드 성공, 테스트 통과, 린터 클린 — 그런데 런타임에서 깨져 있다.

**실제 사례**: Zustand 상태 셀렉터가 매 렌더마다 새 객체 참조를 생성. 결과: 무한 리렌더링, 스트리밍 중 스크롤 깨짐. TypeScript는 문제를 보지 못함. 테스트 통과. 앱은 사용 불가.

**교훈**: 빌드 도구는 구문을 체크합니다. 테스트는 격리된 유닛을 체크합니다. 둘 다 런타임 동작 — 애니메이션 타이밍, 상태 상호작용, 스트리밍 중 스크롤 위치 — 은 체크하지 않습니다. 런타임 검증이 파이프라인의 일급 단계여야 합니다.

### 패턴 2: 에이전트가 자기 규칙을 잊는다

**함정**: 대화 시작 시 스킬 파일을 읽음. 50개 메시지 후, 컨텍스트 압축이 작동하면 규칙이 사라짐.

**실제 사례**: Feature 6 검증 중 컨텍스트 압축이 verify-phases.md 참조를 지움. Feature 1~5에서는 성공적으로 완료했던 Playwright UI 검증을 통째로 건너뜀.

**교훈**: 메타 문제입니다. 모든 다른 안전장치는 에이전트가 지시를 읽는다고 가정합니다. 지시가 컨텍스트에서 압축되면, 아무것도 동작하지 않습니다.

**해결**: 핵심 규칙을 모든 실행 지점에 인라인 (P2). 상태를 파일로 추적 — 에이전트가 규칙을 잊어도 sdd-state.md가 어느 단계에서 재개해야 하는지 알려줌.

### 패턴 3: 에이전트가 읽지 않고 추측한다

**함정**: 소스에서 실제 값을 읽을 수 있는데 그럴듯한 값을 생성.

**실제 사례**: 소스 테마 파일에서 CSS 색상 토큰을 읽는 대신 추측. 요약에서 "사이드바 탭 3개"라고 가정했는데 실제 코드에는 2개 (세 번째는 조건부).

**교훈**: 에이전트는 생성하도록 훈련됩니다. 정답이 `grep` 한 번이면 나오는데도 기꺼이 그럴듯한 답을 생성합니다. 스킬이 소스 읽기를 *강제*해야 합니다.

### 패턴 4: 에러 경로는 물어보기 전에는 존재하지 않는다

**함정**: 스펙이 일이 잘될 때를 기술함. 잘못될 때를 기술하는 사람은 없음.

**실제 사례**: 스트리밍 채팅 Feature의 스펙이 `메시지 전송 → 응답 수신`은 정의했지만 `로딩 → 스트리밍 → 완료 → 에러 → 정리 → 재시도`는 없음.

**교훈**: Brief 상담의 "일이 잘못되면 어떻게 되나요?" 단계가 이 이유로 존재합니다.

### 패턴 5: "하나만 더" 함정

**함정**: 기존 Feature에 작은 요구사항을 추가. 에이전트가 전체 스펙을 재생성하면서 이전에 승인된 SC를 모두 손실.

**해결**: `add --to` augmentation + SC Preservation. 기존 SC는 `[preserved]` 태그, 새 SC는 `[new]` 태그.

## 스킬 개발자를 위한 팁

### 팁 1: 스킬은 가이드가 아니라 계약이다

SKILL.md를 튜토리얼이 아니라 법적 계약처럼 작성하세요. 모든 모호한 표현은 에이전트에게 가장 쉬운 방향으로 해석됩니다.

```
❌ "구현 후 테스트 실행을 고려하세요"
   → 에이전트 해석: "테스트는 선택사항"

✅ "🚫 차단: 테스트 실행. 실패하면 진행 금지.
    다음 단계 전에 실패 테스트를 수정하세요."
   → 에이전트 해석: "테스트를 실행해야 하고 통과해야 한다"
```

### 팁 2: 안티패턴이 패턴보다 중요하다

"올바른" 예시 하나는 참고점 하나. "잘못된" 예시 다음에 "올바른" 예시를 보여주면 경계가 생깁니다.

### 팁 3: 상태 머신이 If/Else를 이긴다

스킬에 여러 상태가 있으면 (pending, in_progress, augmented, completed, regression-specify 등), 파일의 상태 머신으로 모델링하세요 — 스킬 텍스트의 조건 로직이 아니라.

### 팁 4: 컨텍스트 압축을 염두에 두고 테스트하라

스킬 파일은 결국 컨텍스트에서 압축됩니다. 이를 테스트하세요:
1. 대화를 시작
2. 긴 워크플로우 실행 (50+ 메시지)
3. 확인: 에이전트가 여전히 스킬 파일의 규칙을 따르는가?

### 팁 5: 자동화할 수 없으면 위임하라

```
❌ "드래그앤드롭 테스트 건너뜀 (Playwright 제한)"
   → 미테스트 기능이 조용히 통과

✅ AskUserQuestion "파일을 업로드 영역에 드래그해주세요.
   상태가 'idle'에서 'processing'으로 변하나요?"
   → 사용자가 테스트 수행, 결과 보고
```

### 팁 6: 스킬을 소프트웨어처럼 버전 관리하라

history.md에 40개 이상의 항목이 설계 결정을 문서화합니다. 각 항목은 *무엇이* 변했는지뿐 아니라 *왜* — 어떤 실패 패턴이 변경을 촉발했는지 설명합니다.

## 더 큰 그림

spec-kit-skills는 Harness Engineering의 한 시도입니다. 구체적 구현 — 마크다운 파일, 도메인 모듈, HARD STOP — 은 진화할 것입니다. 하지만 근본적 필요는 사라지지 않습니다:

**AI 에이전트가 유능해질수록, 개발자는 그 능력을 방향 짓는 더 구조화된 방법이 필요합니다.**

미래는 "AI가 모든 코드를 쓴다"가 아닙니다. 개발자가 점점 더 정교한 하네스를 만드는 것입니다 — 도메인 전문성을 인코딩하고, 품질 게이트를 강제하며, 장기 실행 워크플로우에서 컨텍스트를 유지하는 시스템.

오늘은 Claude Code 스킬입니다. 내일은 완전히 다른 무언가일 수 있습니다. 하지만 패턴 — Context Continuity, Enforce Don't Reference, File over Memory — 은 사람과 AI 에이전트가 복잡한 소프트웨어에서 협업하는 곳이라면 계속 유효할 것입니다.

```bash
# 설치
git clone https://github.com/coolhero/spec-kit-skills.git
cd spec-kit-skills
./install.sh

# 코드베이스 탐색
/code-explore /path/to/interesting-project

# 새 프로젝트 시작

/smart-sdd pipeline F001
```

## 🤖 에이전트를 위한 글 — 스킬 개발 체크리스트

```yaml
skill_development_checklist:
  structure:
    - SKILL.md: 슬림 라우팅 + 필수 규칙 (항상 컨텍스트에 존재)
    - commands/: 온디맨드 워크플로우 (호출 시 로드)
    - reference/: 보조 규칙 및 스키마
    - SKILL.md는 300줄 이하 유지 — 항상 로드됨

  enforcement:
    - 모든 핵심 규칙: 인라인 지시 + 차단 게이트 + 안티패턴 예시
    - 핵심 동작에 "자세한 내용은 X.md 참조"를 절대 사용하지 말 것
    - HARD STOP: AskUserQuestion 사용, 자동 진행 금지
    - 빈 응답: 항상 재질문, 침묵을 승인으로 해석하지 말 것

  state_management:
    - 모든 파이프라인 상태를 파일에 추적 (에이전트 메모리 아님)
    - 상태 머신 파일 사용 (sdd-state.md 같은)
    - 스키마에 유효한 전환 포함
    - 컨텍스트 압축 후, 에이전트가 상태 파일을 읽어 복구해야 함

  common_failure_patterns:
    build_pass_not_feature_works: 빌드+테스트뿐 아니라 런타임 검증 필요
    agent_forgets_rules: 모든 실행 지점에 인라인 강제
    agent_guesses_not_reads: 능동적 읽기 지시로 소스 파일 읽기 강제
    error_paths_missing: 모든 Brief 상담에 명시적 "잘못될 때" 포함
    context_loss: 압축 후 파일 기반 상태 복구

  tips:
    - 안티패턴 > 패턴 (RIGHT 전에 WRONG을 보여라)
    - 상태 머신 > if/else (상태를 파일에 모델링)
    - 위임 > 건너뛰기 (자동화 실패 시 사용자에게 질문)
    - 버전 이력 > 메모리 (무엇이 아니라 왜를 기록)
    - 인라인 > 참조 (무시되는 참조 1개보다 반복 30개)
```
